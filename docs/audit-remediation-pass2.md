- **Never invent or "fix up" vector bytes.** Expected bytes come from hex in

## P1 ‚Äî x509gost (**Opus**, Wave 1)

Findings: **R2-X509-01** (medium), **R2-X509-02** (low), **R2-X509-03** (low). The
security-sensitive lane: GOST chain building and signature verification. Read
`x509gost/verify.go` fully before editing. Dismissed and off-limits in this package:
`R2-X509-D1`, `R2-X509-D2`.

1. **R2-X509-01 ‚Äî enforce RFC 5280 path-validation on the GOST chain (the one MEDIUM).**
   The GOST branch (`buildGOSTChain`/`signerOf`/`issuerIsCA`, verify.go:207-341) enforces only
   DN match, validity, IsCA, KeyUsageCertSign, and the GOST signature ‚Äî it never consults
   `pathLenConstraint`, unhandled critical extensions, or name constraints, so a GOST leaf gets
   weaker validation than a non-GOST leaf through the same `Certificate.Verify`. Add, per
   issuer link in the chain walk, the checks the stdlib applies (`$GOROOT/src/crypto/x509/
   verify.go:444`, `:500-502`, `checkNameConstraints` are the ground-truth references named in
   the appendix):
   - **Unhandled critical extensions:** reject any cert in the chain whose
     `Stdlib.UnhandledCriticalExtensions` is non-empty (RFC 5280 ¬ß4.2 MUST-reject). Surface a
     named sentinel error (e.g. `errUnhandledCriticalExt`).
   - **pathLenConstraint:** when an intermediate has `BasicConstraintsValid && MaxPathLen >= 0`,
     enforce it against the number of intermediates below it in the chain (stdlib's
     `numIntermediates > c.MaxPathLen` rule).
   - **Name constraints:** process `PermittedDNS`/`ExcludedDNS` (and the other constraint
     fields the stdlib's `checkNameConstraints` covers) on CA certs against the leaf's SANs.
     If full name-constraint processing balloons past ~150 lines, land the critical-extension
     and pathLen checks first (commit + findings line), then report ‚Äî do **not** half-land
     partial name-constraint logic that silently passes some constraint types.
   - Keep the "permissive-when-unset" posture (verify.go:205-206) for *absent* extensions; only
     *present* constraint data must now be honored.
   - **Tests** (`verify_paths_test.go`, using the module's own cert builder): (a) a chain whose
     intermediate carries `pathLenConstraint=0` must reject a 3-deep chain; (b) a cert carrying
     an unrecognized critical extension must be rejected; (c) a name-constrained CA must reject
     an out-of-scope leaf and accept an in-scope one. Each must currently *pass* (bug present)
     and *fail* after the fix is reverted ‚Äî i.e. they pin the new behavior.
   - **Doc:** update the `Verify`/`VerifyOptions` doc comments to state that GOST chains now
     enforce these checks (the appendix notes the current doc doesn't disclose the skip).

2. **R2-X509-02 ‚Äî honor anyExtendedKeyUsage asserted by the leaf.** `leafSatisfiesKeyUsages`
   (verify.go:189-199) checks the *requested* side for `ExtKeyUsageAny` but never the *leaf's
   own* EKU set. Mirror stdlib `checkChainForKeyUsage`
   (`$GOROOT/src/crypto/x509/verify.go:1003-1008`): before the per-request loop, if
   `slices.Contains(leaf.ExtKeyUsage, x509.ExtKeyUsageAny)` return true. Impact today is
   false-rejection only. Test: a GOST leaf with `ExtKeyUsage=[ExtKeyUsageAny]` verifies under
   `KeyUsages=[ServerAuth]` (currently rejected with `errLeafEKUMismatch`); the existing
   specific-EKU cases (`TestVerify_GOST_KeyUsages`) stay green. Update the function doc comment
   (lines 175-178) to list the leaf-asserts-Any passing case.

3. **R2-X509-03 ‚Äî don't require a pool-matched leaf to be self-signed.** In `buildGOSTChain`
   (verify.go:233-247), a leaf byte-identical (`certEqual`) to a `GOSTRoots` entry currently
   runs `verifyGOSTSignature(leaf, leaf)` and `return nil, err` on failure ‚Äî aborting the whole
   search even when a valid chain via a different root exists. Align with stdlib's trust-anchor
   model (a pool member is trusted by provision, no self-signature check ‚Äî
   `crypto/x509/verify.go:596-598`): when the leaf matches a root entry, **accept it as a
   one-element chain** (or fall through to the recursive search) rather than aborting on a
   self-signature failure. Preserve the common self-signed-root happy path
   (`TestVerify_GOST_SelfSigned_Happy`). Test: a non-self-signed (cross-signed) CA placed in
   `GOSTRoots` while also being the leaf, with a *valid* alternate signer root in the pool, must
   verify. Decide and document whether a pool match is trusted unconditionally (stdlib parity)
   or still requires *a* valid signer in the pool ‚Äî pick the stdlib-parity option unless an
   existing test forbids it, and note the choice in the commit body.

**Parity note:** these are GOST-path-only validation additions (no primitive output change), so
the gostcrypto-compat parity gate should be unaffected ‚Äî but run it anyway; if a parity fixture
now fails because it relied on the over-permissive chain behavior, the *fixture* encodes the bug
and the appendix evidence is authoritative ‚Äî report rather than weaken the new check.

**Acceptance:** `go test ./x509gost/` green (incl. `go test -run Fuzz -fuzztime 10s
./x509gost/` smoke); full-module tests green; parity gate green. 2‚Äì3 commits (path-validation /
EKU+self-signed / tests).

## P2 ‚Äî vko + gost3410curves (Sonnet, Wave 1)

Findings: **R2-VKO-01** (vko, robustness) and **R2-CURVE-01** (gost3410curves, doc-only). Paired
because both concern the elliptic-curve cofactor and the guide bug (`R2-CURVE-01`) is the exact
landmine that would corrupt VKO if "fixed" wrong ‚Äî fixing both together closes the loop. The two
findings touch disjoint files (`vko/vko.go`, `gost3410curves/gost3410-curves.md`).

1. **R2-VKO-01 ‚Äî reject unsupported cofactors instead of silently clamping.** `cofactor()`
   (vko.go:137-143) returns 4 only for `Cofactor==4` and **1 for every other value** (0, 2, 8),
   so a caller-supplied curve with a genuine cofactor of 2 or 8 yields a silently wrong KEK.
   No registered GOST curve has such a cofactor (all are `{1,4}`), so this is a hardening fix:
   make `agreementRaw`/the `KEK*` entry points **return an error** (the VKO functions already
   return `([]byte, error)`) when `c.Cofactor` is neither 1 nor 4 ‚Äî e.g. a sentinel
   `errUnsupportedCofactor` naming the value ‚Äî rather than proceeding with the wrong multiplier.
   Keep the `Cofactor==4` and `Cofactor‚àà{0,1}` paths exactly as they compute today (do **not**
   change output for any registered curve ‚Äî that is parity-critical). Test: build a `Curve`
   with `Cofactor: 2` and assert `KEK2012256` returns the error; assert every registered curve
   (tc26-256-A through Curve2001Test) still produces its existing KEK byte-for-byte (a
   regression guard over the current vectors).

   > Caution: cofactor 0 currently maps to 1 and several registered curves set `Cofactor: 1`
   > explicitly ‚Äî confirm via `gost3410curves/curves.go` whether any registered curve carries
   > `Cofactor: 0` (the appendix says they are all `{1,4}`). If any is 0, treat 0 as 1 (current
   > behavior) and reject only `{2,8,‚Ä¶}`; if none is 0, you may reject 0 too. Do not change the
   > KEK output of any curve that has a reference answer.

2. **R2-CURVE-01 ‚Äî correct the stale cofactor instructions in the guide.** The shipped code is
   correct (`curves.go:161-171` sets CryptoPro-C / tc26-256-D to `Cofactor: 1`, which the Hasse
   bound and gost-engine `gost_params.c:52-59` both confirm). The *guide* is wrong at
   `gost3410-curves.md:144`, `:324-336`, and `:430-432`, telling implementers to set cofactor 4
   for CryptoPro-C / tc26-256-D. Edit exactly those lines to say **cofactor 1** for CryptoPro-C
   and tc26-256-D, keeping the cofactor-4 designation for the genuinely-h=4 curves (tc26-256-A,
   tc26-512-C). This is doc-only ‚Äî touch no `.go` file in this step. Cross-check the corrected
   list against `curves.go` so the guide and code agree curve-for-curve.

**Parity note:** R2-VKO-01 adds an error path on inputs no registered curve produces, so parity
fixtures (which use registered curves) must be unchanged ‚Äî run the gate to confirm.

**Acceptance:** `go test ./vko/` green incl. the new reject + regression tests;
`go build ./gost3410curves/` green; full-module tests green; parity gate green. 1‚Äì2 commits
(`vko: reject unsupported cofactors` / `gost3410curves: fix stale cofactor guidance in guide`).

## P4 ‚Äî facade: exports.go / modes.go (**Opus**, Wave 1)

Findings: **R2-SIGN-01** (nonce LE/BE), **R2-FACA-01** (nil-error erasure),
**R2-FACA-02** (zero-check before mod-q). All three live in the root facade
(`exports.go`, `modes.go`); this lane owns those files. The underlying primitives
(`gost3410sign`, `vko`) are **correct and KAT-pinned ‚Äî do not touch them.**

1. **R2-FACA-01 ‚Äî surface the degenerate-key failure.** `PublicKeyRawFromPrivate`
   (exports.go:164) and `PublicKeyRawFromPrivate2001Test` (exports.go:241) call
   `gost3410sign.PublicKeyRaw(...), nil`, hardcoding a `nil` error and erasing the subpackage's
   `nil`-on-degenerate signal (it returns `nil` when the key reduces to 0 mod q, or on
   `IsInfinity`). Change both wrappers to return a non-nil error (e.g. a package sentinel
   `errDegeneratePrivateKey`) when `PublicKeyRaw` yields `nil`. Test: feed `make([]byte, 32)`
   (zero key) and assert `err != nil` and `pub == nil` for both functions; a valid key still
   returns `(pub, nil)`.

2. **R2-FACA-02 ‚Äî move the zero check after mod-q in GenerateEphemeralKey.** `modes.go:291-299`
   runs `d.Sign() == 0` *before* `d.Mod(d, q)`, so an input of `LE(q)` passes the check, reduces
   to 0, and the function returns an all-zero private key with `pub=nil` and `err=nil` ‚Äî while
   the gogost-backed compat facade *errors* on the same input (so the backends diverge). Fix
   per the appendix: keep the read/reduce order that mirrors gogost's `NewPrivateKeyLE`, but add
   a **post-mod** zero check (and/or check `pub == nil`) that returns `errEphemeralZeroKey`.
   Test: feed `raw = LE(q)` for `GOST2001TestParamSetCurve` and assert an error (currently
   returns a zero key + nil pub + nil err); the normal path is unaffected. If
   `FuzzGenerateEphemeralKey` exists, leave its `len(pub) != 2*ps` assertion ‚Äî the fix makes the
   `LE(q)` corner return cleanly.

   > **Parity note (read before coding):** this aligns the clean-room facade's error behavior
   > with gogost on the `LE(q)` input ‚Äî a *convergence*, so it should make the parity gate
   > happier, not break it. Confirm with `( cd ../gostcrypto-compat && go test ./... )`.

3. **R2-SIGN-01 ‚Äî resolve the nonce LE/BE convention mismatch.** The facade draws `k` via
   `rand.Int` then serializes it **little-endian** (`big2leFixed`, exports.go:183), but
   `gost3410sign.SignDigest` reads `k` **big-endian** (its documented contract, KAT-pinned by
   `TestKAT_SignDeterministic`), so the effective nonce is `reverse(k) mod q`. It is **benign**
   (still uniform in `(0,q)`, signatures valid, no bias) ‚Äî so the safe fix is the one that
   doesn't change valid-input output:
   - **Default (recommended): document, don't change behavior.** Add a comment at exports.go:183
     explaining the deliberate LE-write/BE-read reversal (it is a byte permutation of a
     full-width uniform draw, entropy-preserving), and fix the misleading `big2leFixed` comment
     (exports.go:201-203) that calls LE "the nonce encoding". No output change ‚Üí parity gate and
     any seeded signature test stay green.
   - **Only if the team prefers the convention aligned:** serialize `k` big-endian to match the
     `SignDigest` contract. **This changes which nonce a given `rnd` stream produces**, so any
     test that pins a signature for a fixed/seeded `rnd` will change and must be regenerated, and
     the parity gate (if it compares facade signatures under a shared seed) will flag it. Do this
     **only** on explicit instruction; otherwise take the documentation option.

   Pick the documentation option unless the lane prompt says otherwise; note the choice in the
   commit body.

**Acceptance:** `go test ./` (root facade package) green incl. the two new error tests;
full-module tests green; parity gate green. 1‚Äì2 commits (`gostcrypto: surface degenerate-key
errors in facade` / `gostcrypto: document nonce encoding convention`).
