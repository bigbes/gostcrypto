### [CTRA-02] XORKeyStream silently corrupts on inexact dst/src overlap instead of panicking (stdlib panics)

- **Location:** `ctracpkm/ctracpkm.go:139`
- **Category:** robustness · **Severity:** minor · **Confidence:** 75

**Finding:** cipher.Stream's contract says 'dst and src must overlap entirely or not at all'. The stdlib enforces this: crypto/cipher's ctr.XORKeyStream panics with 'invalid buffer overlap' via alias.InexactOverlap. This implementation only checks len(dst) < len(src) and then processes byte-by-byte (dst[i] = src[i] ^ gamma[num]), so a partially-overlapping call silently produces garbage. Probe-confirmed: XORKeyStream(buf[1:], buf[:32]) returns without panic and yields output diverging from the correct keystream XOR (each src byte read after dst[i-1] already clobbered it). For a crypto package, misuse that silently emits wrong ciphertext is worse than a panic. Fix: add the equivalent of an InexactOverlap check (e.g. compare &dst[0]/&src[0] extents) at the top of XORKeyStream; add a guard test asserting the panic.

**Verifier confirmation:** Confirmed by code inspection and a reproduction probe. ctracpkm.go:139-166 (XORKeyStream) checks only len(dst) < len(src) (panic errShortDst, line 140-142) and then writes dst[i] = src[i] ^ c.gamma[c.num] byte-by-byte with no aliasing check — nothing equivalent to crypto/internal's alias.InexactOverlap exists anywhere in the repo (repo-wide grep for "overlap" finds only a doc comment in gost28147cnt/cnt.go:92, no enforcement; no guard test in ctracpkm/guard_test.go covers it; TODO.md and ctr-acpkm.md do not document this as an accepted divergence). A /tmp probe program reproduced both halves of the claim: stdlib cipher.NewCTR(aes).XORKeyStream(buf[1:], buf[:32]) panics with "crypto/aes: invalid buffer overlap", while ctracpkm.NewCTR(kuznyechik).XORKeyStream(buf[1:], buf[:32]) returns without panic and produces output diverging from the correct keystream XOR from byte 1 onward (got 98543f6b... vs want 98cc6b54...), because each src byte after index 0 is read after the previous dst write already clobbered it. The cipher.Stream doc states "dst and src must overlap entirely or not at all" and the stdlib enforces it with a panic; this implementation silently emits wrong ciphertext on the same misuse. Minor robustness finding, accurately categorized and severity-rated.


### [CTRA-03] NewCTRACPKM accepts block sizes that do not divide the 32-byte ACPKM key; panics with slice-bounds error mid-stream at the first rekey instead of at construction

- **Location:** `ctracpkm/ctracpkm.go:182`
- **Category:** robustness · **Severity:** minor · **Confidence:** 80

**Finding:** rekeyACPKM iterates `for off := 0; off < 32; off += n { c.block.Encrypt(newKey[off:off+n], acpkmD[off:off+n]) }`. If the caller's newBlock returns a Block with BlockSize that doesn't divide 32 (e.g. 64, or 12), the constructor's checks all pass (len(iv)==bs, sectionSize%bs==0 is satisfiable, key is 32 bytes), and the failure surfaces only after a full section has been emitted, as an uncontrolled 'slice bounds out of range' panic inside rekeyACPKM — i.e. data may already have been consumed/transmitted. RFC 8645 §6.2.1 defines J=ceil(k/n) with MSB_k truncation, so non-dividing block sizes are spec-conceivable; this implementation supports only k%n==0 (correct for Magma/Kuznyechik) but should enforce `acpkmKeySize%bs == 0` (and arguably bs<=32) in NewCTRACPKM with a clear panic message, matching the package's fail-at-construction philosophy (cf. the key-length panic). Not reachable with the in-repo GOST ciphers; only with a foreign cipher.Block via the public factory parameter. Relatedly, a nil newBlock also produces a raw nil-deref at construction rather than a descriptive panic.

**Verifier confirmation:** Reproduced exactly as claimed. NewCTRACPKM (ctracpkm/ctracpkm.go:96-116) checks key length, IV length, and sectionSize%bs==0 but never that bs divides the 32-byte ACPKM key; rekeyACPKM (ctracpkm.go:182-184) slices acpkmD[off:off+n] assuming exact division. A /tmp repro with a 12-byte-block cipher.Block (sectionSize=24, all constructor checks pass) emitted a full 24-byte section, then panicked mid-stream at the first rekey with 'slice bounds out of range [:36] with capacity 32' — an uncontrolled runtime panic after data was already produced, contradicting the package's fail-at-construction style (cf. the descriptive key-length panic at line 104, covered by guard_test.go). Not a documented divergence (TODO.md's ACPKM entry concerns MAC vectors only); ctr-acpkm.md:122 only implicitly scopes to GOST block sizes. Severity minor is accurate: unreachable with in-repo Magma/Kuznyechik, reachable only via a foreign cipher.Block through the public newBlock factory parameter.

#### DISMISSED: No fuzz targets in the package; streaming split-invariance and plain-CTR differential vs stdlib are unfuzzed

- **Location:** `ctracpkm/ctracpkm_test.go` · **Category:** fuzz-gap · **Severity claimed:** minor

**Original claim:** There are zero Fuzz* targets (the only fuzz target in the module is x509gost). The .md guide itself prescribes a differential fuzz harness for this primitive (ctr-acpkm.md §'Differential fuzz harness'), but the gogost-based version can't exist (gogost ships no CTR/ACPKM). Two concrete in-module targets are the right tool here: (1) FuzzCTRACPKM_SplitInvariance — random 32-byte key, nonce, small block-aligned sectionSize ((n%8+1)*bs), arbitrary plaintext, and fuzz-chosen split points; property: concatenated XORKeyStream calls == one-shot output, plus encrypt/decrypt round-trip — this fuzzes exactly the partial-gamma-carry + rekey-at-block-boundary interaction that TestCTR_PartialBlock pins only at 5 fixed split points and only for Kuznyechik/section=32; (2) FuzzCTR_MatchesStdlibCTR — plain NewCTR over kuznyechik.NewCipher/magma.NewCipher differentially against crypto/cipher.NewCTR with the same Block and full-block IV: the two constructions are semantically identical (stdlib CTR also increments the whole counter block big-endian, last byte first), giving a true external oracle for the plain-CTR streaming path at zero dependency cost.

**Why dismissed:** The finding's key premise is wrong: the .md-prescribed differential fuzz harness exists as FuzzDiff_CTRACPKM_vs_Oracle (gostcrypto-compat/parity/ctracpkm/ctracpkm_parity_test.go:83), built on an in-repo gost-engine-traced oracle (gostcrypto-compat/ctr_gost.go) precisely because gogost lacks CTR-ACPKM. It fuzzes cipher choice, key, IV, section size 0-4096 (0 = plain CTR), and plaintext up to 16 KiB — the proposed FuzzCTRACPKM parameter space. Per the workspace parity-gate design, differential fuzzing of gostcrypto primitives intentionally lives in gostcrypto-compat, not in the BSD module. Proposed FuzzCTR_MatchesStdlibCTR is redundant: plain CTR already has an external oracle (TestDiff_PlainCTR_vs_Oracle, parity test :131; fuzz at section=0; in-module TestCTRACPKM_MatchesPlainCTR_WhenSectionZero at ctracpkm_test.go:218). The only unfuzzed sliver — randomized split points — is immaterial: XORKeyStream (gostcrypto/ctracpkm/ctracpkm.go:139-167) is a byte-at-a-time loop with no per-call bulk path, so split invariance holds by construction, and the carried-state/rekey-boundary interaction is already pinned by TestCTR_PartialBlock in both modules (splits crossing the section=32 rekey boundary mid-partial-block). All cited tests verified green.

### [MGM-51] maxFieldLen returns -1 on 32-bit platforms, making Kuznyechik-MGM unusable (every Seal panics, every Open fails)

- **Location:** `mgm/mgm.go:200-209`
- **Category:** correctness · **Severity:** major · **Confidence:** 90

**Finding:** maxFieldLen guards the 1<<(n/2-3) shift with `if bits-lengthFieldShiftAdj >= maxFieldShiftBits` where maxFieldShiftBits is hardcoded 63 — the comment even says 'bit width of a signed 64-bit int'. No test covers 32-bit (nothing reasonably can per-arch, but a `GOOS=linux GOARCH=386 go vet ./...` CI leg would have caught the build at least via a constant-overflow variant; an explicit UintSize-aware unit test of maxFieldLen is portable).

**Verifier confirmation:** Confirmed by reproduction. mgm/mgm.go:205 guards the shift with hardcoded maxFieldShiftBits=63 (mgm.go:49, comment says 'bit width of a signed 64-bit int'); for Kuznyechik bits-3=61<63, so mgm.go:209 computes (1<<uint(61))-1 as a non-constant shift on platform int. **Coverage assessment:** Well-pinned: malformed-DER rejection (3 unit cases + FuzzParseCertificate with 6 seeds, clean 30s smoke), GOST-2012/256 self-signed parse+verify happy path, tampered-signature and wrong-signer negatives, expiry, hostname mismatch, stdlib delegation (RSA), real RSA-signed/GOST-SPKI fixture certs (2012-256 and 2001), CurveByOID resolution for all 10 curve OIDs, and SPKI/PKCS#8-style PEM/DER round-trips for all 10 curves. ### [X509-69] extractGOSTPubKeyBytes OCTET-STRING/raw ambiguity and no pubkey length validation at parse time
