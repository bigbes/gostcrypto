# Audit remediation plan

Execution plan for the 82 confirmed findings of the 2026-06-10 per-package audit.
Companion document: **`docs/audit-findings.md`** — the full machine-generated findings
reference. Every task below cites finding IDs (e.g. `MGM-51`); the implementer MUST
read the cited appendix sections before coding — they contain exact locations,
reproduction evidence, vector hex, and verifier confirmation that the issue is real.

This plan is written to be executed by subagents. Each **lane** is one subagent task.
The default implementer model is **Sonnet**; three lanes are marked **Opus** because
they involve security-semantics decisions or a cipher-core rewrite. The orchestrator
dispatches lanes per the wave schedule in §2.

## 1. Global rules — every implementer reads this first

These rules are binding for every lane. Violating any of them is a hard failure.

1. **Read `CLAUDE.md` in the repo root** (gostcrypto module) before starting.
2. **Clean-room constraint (license-critical).** This module is BSD-2-Clause and must
   never contain code derived from gogost (GPL) or any GPL source. When a fix needs
   lookup tables or constants, **generate them programmatically at `init()` from the
   already-existing clean-room code or compute them from the spec** — never copy a
   table from gogost, and never open gogost source for "reference" while writing code.
   `tmp/engine/` (gost-engine, Apache-2.0) may be read for *test vectors only*, cited
   as `tmp/engine/<path>:<line>`.
3. **No new dependencies.** `go.mod` stays zero-dep. No `golang.org/x/...` either.
4. **Build/test commands** (the harness sandbox blocks the Go build cache — pass
   `dangerouslyDisableSandbox: true` on Bash calls running go build/test/vet):
   ```sh
   CGO_ENABLED=0 go build ./...          # must stay green module-wide
   CGO_ENABLED=0 go vet ./...
   CGO_ENABLED=0 go test ./<your-package>/   # during the lane
   CGO_ENABLED=0 go test ./...               # before your final commit
   ```
5. **The parity gate.** Any change to primitive *behavior* or any *exported API*
   change must be re-validated against the gogost reference:
   ```sh
   ( cd ../gostcrypto-compat && go test ./... )
   ```
   If your lane changes an exported signature (only lanes KEG and CNT do), you must
   also update the call sites in `../gostcrypto-compat` (it is GPL — you may edit its
   *call sites and test wiring*, but never move code from it into gostcrypto).
6. **Vectors are extracted, never invented.** Every new KAT byte string must carry a
   citation comment: `tmp/engine/<path>:<line>`, `rfc/<file>.txt:<line>` (several
   packages bundle their RFC/standard under `<pkg>/rfc/`), or the audit appendix ID
   when the appendix already contains verified hex (the appendix hex was empirically
   confirmed against the implementation by the audit's verifier agents — cite the
   original source named there, e.g. "R 1323565.1.017-2018 Appendix A.1").
7. **Known divergences are not bugs.** Re-read `TODO.md` (root) before "fixing"
   anything related to S-box row order, R 34.11-94 empty-input finalization, or
   CryptoPro key meshing.
8. **Dismissed findings are off-limits.** `docs/audit-findings.md` lists dismissed
   findings per package. Do not implement them, even if they look reasonable.
9. **Panic style.** Validation panics use package-prefixed messages, e.g.
   `panic("ctracpkm: block size must divide the 32-byte ACPKM key")`. Functions that
   already return errors get wrapped sentinel errors instead (match the package's
   existing style — look at its current code first).
10. **Commits.** One lane = 1–3 logical commits. Subject `<package>: <imperative summary>`.
    Body ends with the finding IDs line: `Findings: CTRA-01, CTRA-02`. Never add
    `Co-Authored-By`/`Generated by` trailers.
11. **Scope discipline.** Fix what the cited findings say. No drive-by refactors, no
    style sweeps, no touching other packages except where a task explicitly says so.
12. **Doc edits** (the `<pkg>/<primitive>.md` guides): the stale sections all describe
    the pre-split monorepo ("Status: gogost-backed", `go.bigb.es/tlsdialer` paths,
    build tags that no longer exist). Rewrite those sections to describe *this*
    module truthfully: clean-room implementation lives here, tests are in
    `<pkg>/*_test.go`, parity tests live in `../gostcrypto-compat/parity/<pkg>/`.
    Keep the spec/algorithm sections (the valuable part) intact unless a finding says
    they are wrong.

### 1.1 Anti-footgun rules — read twice, these are where implementations go wrong

- **Never invent, derive-from-memory, or "fix up" vector bytes.** Every expected
  byte string comes from: (a) hex embedded in this plan or in
  `docs/audit-findings.md` (already verified), (b) a cited file in the repo
  (`rfc/*.txt`, `tmp/engine/...`, `testdata/...`), or (c) a tool run whose exact
  command line you record in the test comment. No other source exists.
- **When a new KAT fails, the expected bytes are presumed right and your test
  wiring wrong.** Debug order: (1) byte order — GOST wire format is little-endian,
  RFC tables are big-endian; (2) S-box choice; (3) parameter order; (4) re-read
  `TODO.md` for a documented divergence. If still failing: STOP and report. Do
  NOT edit the expected bytes to match the implementation, and do NOT change
  primitive code to make a new test pass.
- **Never weaken, skip, or delete an existing assertion to get green.** If an
  existing test conflicts with your change, your change is wrong or the lane spec
  is wrong — stop and report which.
- **Behavior changes are only the ones this plan names.** New validation
  panics/errors on *invalid* input are in scope; any change to output bytes for
  *valid* input is out of scope everywhere except where the plan explicitly says
  so, and is always proven by the parity gate.
- **Don't export new symbols** unless the lane says so; don't add public API "for
  testability" — use internal tests (`package <pkg>` test files) to reach
  unexported code.
- **If this plan contradicts the code you read** (a cited line moved, a named
  symbol doesn't exist), the code wins — report the discrepancy in your final
  message; don't guess at intent.
- **The fix for a lying comment is to make it tell the truth**, not to change the
  code to match the lie (relevant to CTRA-01, GOST-17, STRE-59, FACA-78).

## L0 — Foundation (Sonnet, Wave 0)

Tiny enabling commit so Wave-1 lanes don't collide.

**Step 1 — `internal/alias` package.** Create `internal/alias/alias.go` with the
standard aliasing predicates (write them fresh — this is ~15 lines of pointer
arithmetic, original code, do not copy from the stdlib's internal package):

```go
// Package alias provides dst/src overlap checks for streaming primitives.
package alias

// AnyOverlap reports whether x and y share any memory.
func AnyOverlap(x, y []byte) bool {
    return len(x) > 0 && len(y) > 0 &&
        uintptr(unsafe.Pointer(&x[0])) <= uintptr(unsafe.Pointer(&y[len(y)-1])) &&
        uintptr(unsafe.Pointer(&y[0])) <= uintptr(unsafe.Pointer(&x[len(x)-1]))
}

// InexactOverlap reports whether x and y overlap without x[0] == y[0].
// cipher.Stream/Block implementations must panic on inexact overlap.
func InexactOverlap(x, y []byte) bool {
    if len(x) == 0 || len(y) == 0 || &x[0] == &y[0] {
        return false
    }
    return AnyOverlap(x, y)
}
```
Add a small `alias_test.go` (exact overlap, inexact overlap, disjoint, empty).

**Step 2 — `SBox()` accessor on `gost28147.Cipher`.** Add to `gost28147/gost28147.go`:
`func (c *Cipher) SBox() SBox` returning the S-box the cipher was constructed with
(store it on the struct if it isn't already). One-line test pinning it. This enables
L7's `NewCNT` signature fix.

**Acceptance:** `CGO_ENABLED=0 go build ./... && go vet ./... && go test ./internal/alias/ ./gost28147/` green. One commit: `internal/alias, gost28147: add overlap checks and SBox accessor`.

## L1 — mgm (Sonnet, Wave 1)

Findings: **MGM-51, MGM-52, MGM-53, MGM-54, MGM-55, MGM-56** (read all six in the appendix).

1. **MGM-51 (the one real correctness bug of the module — do this first).**
   `mgm/mgm.go:200-209`: `maxFieldLen` hardcodes `maxFieldShiftBits = 63`; on 32-bit
   platforms (`int` = 32 bits) the Kuznyechik case computes `(1 << 61) - 1` on `int`,
   which Go defines as `0 - 1 = -1` → every Seal panics, every Open fails.
   Fix: compare against the platform width using `math/bits.UintSize`:
   when `bits - lengthFieldShiftAdj >= bits.UintSize - 1`, return `math.MaxInt`
   (i.e. the cap doesn't fit an int, so the practical limit is "any int length");
   otherwise return `(1 << shift) - 1` as today. Add a portable unit test of
   `maxFieldLen` that asserts the function never returns a negative value for both
   block sizes regardless of `bits.UintSize`, plus the exact expected values on
   64-bit. Add a cross-compile smoke check to your verification:
   `GOOS=linux GOARCH=386 CGO_ENABLED=0 go build ./mgm/`.
2. **MGM-55.** Enforce RFC 9058's combined bound: validate
   `bitlen(A) + bitlen(P) < 2^(n/2)` (sum, in bits) in both Seal and Open, in
   addition to the per-field cap (which protects the length-block packing). Use
   overflow-safe arithmetic (`uint64`). Cite `rfc/rfc9058.txt:281-282` in the check's
   comment. Negative test with Magma and two fields that individually pass but
   jointly violate — construct lengths arithmetically; do NOT allocate gigabyte
   buffers: factor the check into a small function (e.g. `validateLens(aLen, pLen int)`)
   and unit-test that function directly.
3. **MGM-52.** Port RFC 9058 Appendix A Example 2 (A.1.2 Kuznyechik and A.2.2 Magma):
   the empty-plaintext / AAD-only and exact-block-multiple cases. Vectors are in the
   package's bundled `mgm/rfc/rfc9058.txt` — extract with line citations. Test both
   Seal output (ciphertext||tag) and Open round-trip, plus Open failure on a flipped
   tag bit for each.
4. **MGM-53.** End-to-end tests for truncated tags: for S in {4, 8, 12} (Kuznyechik)
   and {4, 6} (Magma): Seal/Open round-trip succeeds, corrupted tag fails, and the
   tag equals the full-size tag's prefix (MSB truncation per RFC).
5. **MGM-54.** Add `FuzzSealOpenRoundTrip` (random key/nonce/aad/plaintext sizes →
   Seal → Open must return the plaintext; mutate one byte of ciphertext → Open must
   fail) and an in-place aliasing test (`Seal(pt[:0], ...)` over the plaintext buffer).
   Seed the corpus from the existing KATs.
6. **MGM-56.** Fix `mgm-aead.md` per the finding: stale "gogost-backed" status, wrong
   data-limit statements (align with the now-enforced RFC bound), stale symbol name.

**Acceptance:** all new tests green; `GOARCH=386` build green; parity gate green
(`cd ../gostcrypto-compat && go test ./parity/mgm/ ./...` — behavior change is the
new validation only; if a parity test feeds inputs now rejected by the combined-bound
check, the bound is enforced wrong — recheck before weakening anything).

## L2 — x509gost (**Opus**, Wave 1)

Findings: **X509-65, X509-66, X509-67, X509-68, X509-69, X509-70, X509-71, X509-72**.
This is the security-sensitive lane: chain building and signature verification.
Read `x509gost/verify.go` and `x509gost/parse.go` fully before editing.

1. **X509-65 — hash from the signature OID, not the pubkey OID.**
   `verifyGOSTSignature` (verify.go:205) must derive the digest from the **child
   certificate's signature algorithm OID** (RFC 9215: the signwithdigest OID dictates
   the digest over TBSCertificate). Keep `GOSTAlgo` (pubkey OID) for key parsing /
   KEX roles. There is already OID→algo mapping for signature OIDs in `oids.go` /
   `parse.go` — reuse it (`gostAlgoFromSigOID` per the finding). Regression test:
   build a mixed-strength chain with the module's own builder (512-bit CA signing a
   cert carrying a 256-bit key, signed with signwithdigest-512) — it must verify;
   and the inverse (256 CA → 512-key leaf). The audit's verifier already reproduced
   this exact scenario; the appendix section describes the probe.
2. **X509-66 — skip non-GOST roots.** In `buildGOSTChain` (verify.go:156), a non-GOST
   entry in `GOSTRoots` must be **skipped**, not abort the search. Return the
   mixed-chain error only if the loop completes with no GOST candidate at all (or
   drop that error in favor of `errNoValidGOSTChain` — pick whichever keeps existing
   error-message tests green and document the choice). Test: valid chain with
   `GOSTRoots = [rsaCert, gostCA]` (was failing) and `[gostCA, rsaCert]` (was
   passing) — both must verify; also pin the self-signed + mixed-pool case.
3. **X509-67 — enforce KeyUsages on the GOST path.** The GOST branch of `Verify` must
   check the leaf's ExtKeyUsage against `opts.KeyUsages` with stdlib semantics:
   empty opts ⇒ `ExtKeyUsageServerAuth` is NOT implied — match what the doc comment
   promises; if the doc says "the set of extended key usages the leaf must satisfy",
   implement: leaf passes if `opts.KeyUsages` is empty, contains `ExtKeyUsageAny`,
   the leaf has no EKU extension (unconstrained), or any requested usage is present.
   Additionally check the GOST issuer: `BasicConstraints.IsCA` and
   `KeyUsageCertSign` when those extensions are present (stdlib parity). Tests: GOST
   leaf with EKU=codeSigning must fail `KeyUsages=[ServerAuth]` verification; same
   leaf passes with empty KeyUsages; issuer without IsCA fails.
4. **X509-68 — intermediates.** Two-part fix: (a) document the current depth
   limitation in the public `Verify`/`VerifyOptions` doc comments; (b) implement
   depth-bounded chain building: add `GOSTIntermediates []*Certificate` (mirroring
   stdlib's Intermediates pool) and recurse leaf→intermediate*→root with a max depth
   of, say, 8 and the same per-link checks as 1–3 (signature OID hash, IsCA,
   KeyUsageCertSign). Test: root → intermediate → leaf chain (generate with the
   module's own cert builder, as `certbuilder_test.go` does) verifies; chain missing
   the intermediate fails; depth bound enforced. If (b) balloons beyond ~200 lines,
   stop, commit (a) + findings 1–3, and report — do not half-land chain recursion.
5. **X509-69 — pubkey length validation at parse time.** In
   `extractGOSTPubKeyBytes` (parse.go:308): once `CurveOID` is known, require
   `len(PubKeyRaw) == 2*pointSize` for both the OCTET-STRING and raw branches, and
   only accept the OCTET-STRING parse when the inner length is a plausible key size
   (64 or 128), eliminating the 2⁻¹⁶ misparse. Wrong length ⇒ parse error naming the
   got/want lengths. Tests: 63-byte and 1-byte keys rejected at parse; a crafted raw
   key starting with a valid-looking `0x04 len` header parses via the correct branch.
6. **X509-70 — externally-generated fixture.** All current GOST signature-format
   tests are self-consistent (this module signing for itself). Generate real
   fixtures with OpenSSL 3 + gost-engine:
   `OPENSSL_CONF=/opt/homebrew/etc/gost/gost-engine.cnf /opt/homebrew/opt/openssl@3/bin/openssl req -x509 -newkey gost2012_256 ...`
   for at least: 256-A self-signed, 512-A self-signed, and a 512-CA→256-leaf pair
   (which also exercises X509-65). Commit the PEMs under `x509gost/testdata/certs/`
   plus the generation script `x509gost/testdata/certs/generate.sh` (script is
   documentation; tests must not invoke openssl at runtime). Parse+verify tests over
   the fixtures pin the wire format against an independent implementation.
7. **X509-71 — verify-path tests.** Add: depth-1 chain success (CA-signed leaf, both
   256 and 512), GOST-2001 verify (34.11-94 hash branch), not-yet-valid cert
   rejected, empty `GOSTRoots` rejected. Several of these fall out of tasks 1–6 —
   dedupe, don't double-write.
8. **X509-72 — doc fix.** `parse.go:78`: `PubKeyRaw` byte-order comment contradicts
   its own reference; verify the truth empirically against a fixture from task 6
   (parse a cert whose key you generated, compare X/Y) and fix the comment to match
   reality, citing the fixture.

**Acceptance:** `go test ./x509gost/` green including fuzz seeds

parity gate green. Commit in 2–3 logical commits (verify fixes / parse fixes /

## L3 — kuznyechik (**Opus**, Wave 1)

Findings: **KUZN-44, KUZN-45, KUZN-46, KUZN-47, KUZN-48**.

1. **KUZN-44 — table-driven performance rewrite.** Current throughput ≈0.3 MB/s
   (bit-loop GF multiply, ~2160 multiplies/block); the TLS suite 0xC100 needs this
   as a bulk cipher. Required approach (clean-room-safe):
   - Keep the existing `gf()`, `l()`, S-box and round structure as the *generator*.
   - At `init()` (or lazily, once), precompute the standard fused tables: for
     encryption, `encTable[16][256]` of 16-byte vectors where
     `encTable[pos][b] = L(S(b) at position pos)` — i.e. combine the π S-box and the
     linear transform L into per-byte-position lookups so a round becomes 16 table
     loads XORed together. Same for decryption (`L⁻¹∘π⁻¹`) and for the key schedule
     if it uses L directly. Derive every table entry by *calling the existing
     verified functions* — no literal tables in source beyond what already exists.
   - Keep the slow path code (it generates the tables and documents the math); the
     fast path replaces per-block work only.
   - Add `BenchmarkEncrypt`; target ≥50 MB/s single-core (gogost-class is
     ~100 MB/s; anything ≥50 unblocks TLS use). Record before/after numbers in the
     commit message.
   - Memory note: 2 × 16 × 256 × 16 B = 128 KiB of tables — fine; compute once,
     share across cipher instances (package-level `sync.Once`).
2. **KUZN-48.** Document the (unchanged) timing model: the table-driven version is
   still secret-index lookup based, like the references. Add a short
   `kuznyechik/SECURITY.md` modeled on `gost3410curves/SECURITY.md`: not
   constant-time, cache-timing considerations, and that this matches gogost/engine.
   Reference it from the package doc comment.
3. **KUZN-46.** Test in-place operation: `Encrypt(buf, buf)` / `Decrypt(buf, buf)`
   equal the out-of-place result for the official KAT block.
4. **KUZN-47.** Tests: `NewCipher` copies the key (mutate the caller's slice after
   construction; ciphertext unchanged) and panics on len ≠ 32 (under and over).
5. **KUZN-45.** Guide doc sweep per §1 rule 12 (stale Status/conformance/fuzz
   sections in `kuznyechik-gost34122015.md`).

**Acceptance — this is parity-gate-critical:** all existing KATs green
(GOST R 34.12-2015 vectors must produce identical bytes), `go test ./kuznyechik/`
green, **full parity gate green** (`cd ../gostcrypto-compat && go test ./...` — the
differential tests sweep many input shapes; a table-generation bug will show here),
benchmark evidence in the commit. Also re-run dependent in-module suites:
`go test ./ctracpkm/ ./mgm/ ./omac/ ./kexp15/ ./keg/`.

1. **KDFT-30.** The authoritative vector is bundled at
   `kdftree/rfc/rfc7836.txt:1528-1559` (Appendix B, example 10):
   ```
   K_in  = 000102030405060708090a0b0c0d0e0f101112131415161718191a1b1c1d1e1f
   label = 26bdb878
   seed  = af21434145656378
   ```

   32-byte output is *nothing* — the 32-byte path gets its content coverage from
   KDFT-31's manual-HMAC reconstruction instead.

   for each r in 1..4, cross-check `KDFTree256(key, label, seed, r, 64)` against a
   manual inline reconstruction in the test: for i = 1..2,
   `K(i) = HMAC-Streebog256(key, [i]_b(r bytes, BE) || label || 0x00 || seed || [L]_b(minimal BE, L=512))`
   (the implementation encodes `[L]_b` with leading zeros stripped — kdftree.go
   doc comment; the manual code in the test must encode it independently from the
   spec text, not call the package's helpers). This pins r=2..4 content and
   doubles as the 32-byte-content check when run with outLen=32/L=256.
3. **KDFT-32.** Truncation content (not just length): with fixed inputs and
   outLen=40 (r=1, L=320), assert
