# KDF_TREE_GOSTR3411_2012_256 (KDFTree2012_256)

## What it is

`KDFTree2012_256` is a counter-mode key-derivation function built on
HMAC with the Streebog-256 hash (GOST R 34.11-2012, 256-bit output). It
takes a derivation key plus a `label` and a `seed` and expands them into
an arbitrary multiple of 32 bytes of keying material. It is the KDF defined
in **RFC 7836 §4.4** ("The Key Derivation Function
KDF_TREE_GOSTR3411_2012_256"), also published as Russian recommendation
**R 50.1.113-2016 §4.5**. The single-block variant (RFC 7836 §4.5,
`KDF_GOSTR3411_2012_256`) is just this function with `R = 1` and `L = 256`,
i.e. exactly the first (and only) iteration producing 32 bytes.

*Intended implementer: a Sonnet-class coding agent — every constant, S-box, parameter table, and edge case is inlined so this primitive can be built without consulting gogost or external specs.*

In gost-engine the C entry point is `gost_kdftree2012_256()`
(`tmp/engine/gost_keyexpimp.c:201`). It is a workhorse used by three
distinct higher-level constructions in Tarantool's GOST stack:

- **KEG / key expansion** for RFC 9367 ciphers — `internal/gost/keg_gost.go:77`
  calls `KDFTree2012_256(tmpkey, "kdf tree", ukmSource[16:24], 64)` to split a
  VKO-derived `tmpkey` into a 64-byte `(cipher_key || mac_key)` pair. The
  matching engine path is `gost_gost2015.c:173`
  (`gost_kdftree2012_256(keys, 64, inkey, 32, ...)`).
- **TLSTREE** key meshing — the engine's `gost_tlstree()`
  (`tmp/engine/gost_keyexpimp.c:261`) chains three `gost_kdftree2012_256`
  calls with labels `"level1"`, `"level2"`, `"level3"` and masked TLS
  sequence numbers as seeds. (Our pure-Go TLS path uses gogost's `TLSTree`
  wrapper for this, but the underlying primitive is the same KDF tree.)
- **kexp15 / VKO key transport** — `gost_ec_keyx.c:166`.

### Repository status

`statusKind = clean-room`. This package (`kdftree/kdftree.go`) is a
clean-room implementation of KDF_TREE_GOSTR3411_2012_256, built from the RFC
spec and the implementation notes in this guide. It imports only `crypto/hmac`,
`encoding/binary`, and the sibling clean-room Streebog-256 hash
(`github.com/bigbes/gostcrypto/streebog`). No gogost, no GPL code.

Dedicated tests live in `kdftree/kdftree_test.go`. Differential parity tests
against the gogost reference (which is GPL-3.0) live in the separate
`gostcrypto-compat` module at `gostcrypto-compat/parity/kdftree/`.

The reason this module re-rolls the KDF loop rather than calling gogost's
`KDF.Derive`: gogost's `Derive`
(`third_party/gogost/gost34112012256/kdf.go:31`) **hardcodes** the iteration
counter to the single byte `0x01` (line 32) and the length suffix to the two
bytes `0x01 0x00` (lines 44-47). It therefore only ever emits 32 bytes
(`L = 256` bits) in one HMAC call and cannot produce the 64-byte output the
KEG step needs. See "RFC ↔ implementation deltas" below.

## Specification

### HMAC core (RFC 7836 §4.1.1)

`HMAC_GOSTR3411_2012_256` is plain HMAC (RFC 2104) instantiated with:

- Hash `H_256` = GOST R 34.11-2012 with 256-bit output (Streebog-256, RFC 6986).
- HMAC output size: **32 bytes**.
- HMAC block size (the `ipad`/`opad` width): **64 bytes**.

`ipad`/`opad` are the standard RFC 2104 constants (`0x36`/`0x5c` repeated).
Go's `crypto/hmac` handles all of this given a `func() hash.Hash` for
Streebog-256.

### KDF_TREE (RFC 7836 §4.4)

Inputs:

| Symbol  | Meaning                                                            |
|---------|-------------------------------------------------------------------|
| `K_in`  | derivation key (HMAC key); any length                             |
| `label` | protocol-assigned byte string, fixed length per use site          |
| `seed`  | protocol-assigned byte string, fixed length per use site          |
| `R`     | width in bytes of the iteration-counter field (1, 2, 3, or 4)     |
| `L`     | required output size **in bits**, `L <= 256 * (2^(8*R) - 1)`      |

Output, normative formula (RFC 7836 §4.4):

```
KDF_TREE_GOSTR3411_2012_256(K_in, label, seed, R) = K(1) | K(2) | K(3) | ...

K(i) = HMAC_GOSTR3411_2012_256(K_in, [i]_b | label | 0x00 | seed | [L]_b)
```

where:

- `[i]_b` — byte representation of the iteration counter `i`, **network byte
  order (big-endian)**, exactly `R` bytes wide.
- `0x00` — a single literal zero separator byte between `label` and `seed`.
- `[L]_b` — byte representation of `L` (the bit-length of the *total* desired
  output), **network byte order (big-endian)**, **variable length with no
  leading zero bytes added** (RFC 7836 §4.4: "no leading zero bytes added").
- `|` denotes concatenation.

The number of iterations is `ceil(L / 256)`. Each `K(i)` is a full 32-byte
HMAC block; the concatenation is truncated to `L` bits if `L` is not a
multiple of 256. `KDFTree256` supports arbitrary `outLen > 0` and truncates
accordingly; all in-repo consumers use multiples of 32 bytes.

### Single-block KDF (RFC 7836 §4.5)

```
KDF_GOSTR3411_2012_256(K_in, label, seed)
  = HMAC_GOSTR3411_2012_256(K_in, 0x01 | label | 0x00 | seed | 0x01 | 0x00)
```

This is `KDF_TREE(..., R=1)` with `L = 256`: `[1]_b = 0x01`,
`[256]_b = 0x01 0x00`. Exactly what gogost's `KDF.Derive` hardcodes.

### Parameters used by this repo

The in-module consumers (keg, kexp15, tlstree) always call with:

- `R = 1` (1-byte counter; the `representation` argument is `1` at every
  engine call site — `gost_keyexpimp.c:296-301`, `gost_gost2015.c:173`).
- `outLen` a positive multiple of **32 bytes** (the standard use cases are
  32 and 64). For KEG it is 64.
- `L = outLen * 8`. For 64 bytes, `L = 512`, and `[L]_b = 0x02 0x00`.

`KDFTree256` supports the full generality: `R` in 1..4, any positive `outLen`
(panics if it exceeds what the r-byte counter can address). The only in-repo
consumers use `R = 1` because the engine always uses `R = 1`.

Because `R = 1`, the counter field is a single byte `0x01, 0x02, ...`. With
`outLen <= 32*255 = 8160` bytes the 1-byte counter never overflows.

### Per-iteration HMAC input layout (R = 1)

```
  +------+----------------+------+---------------+----------------+
  | i_b  |     label      | 0x00 |     seed      |     L_b        |
  | 1 B  |  len(label) B  | 1 B  |  len(seed) B  | 1 or 2 B (BE)  |
  +------+----------------+------+---------------+----------------+
   i = 1..ceil(L/256), big-endian
```

`L_b` is the **same value for every iteration** (it encodes the total output
length, not the per-block length). Only `i_b` changes between iterations.

## RFC ↔ implementation deltas

### D1 — gogost `KDF.Derive` is the §4.5 single-block KDF only, NOT the tree

`third_party/gogost/gost34112012256/kdf.go:31-53` implements exactly
`KDF_GOSTR3411_2012_256` (RFC 7836 §4.5): counter byte hardcoded to `0x01`
(line 32) and length suffix hardcoded to `0x01 0x00` (lines 44-47, i.e.
`L = 256`). It produces **32 bytes, once**. It is *not* a multi-iteration
tree and cannot derive 64 bytes. The repo wrapper therefore ignores
`KDF.Derive` and re-rolls the loop
(`internal/gost/kdftree_gost.go:40-48`), changing the counter byte per
iteration and emitting `L_b = 0x02 0x00` for 64-byte output. **A
reimplementer who naively wires up gogost's `Derive` will get the wrong
answer for any output longer than 32 bytes.** (Documented in
`CLAUDE.md` → "gogost/v7 library gotchas" → `KDF.Derive`.)

### D2 — `[L]_b` strips leading zero bytes (engine and this module)

RFC 7836 §4.4 says `[L]_b` carries **no leading zero bytes**. The engine
enforces this literally: `gost_keyexpimp.c:212` computes
`len_repr = be32(keyout_len * 8)` as a 4-byte big-endian integer, then
`gost_keyexpimp.c:227-231` walks `len_ptr` forward while the leading byte is
zero, shrinking `len_repr_len` from 4 down to the minimal width.

This module (`kdftree/kdftree.go:102-113`, `encodeNoLeadingZeros`) implements
the same leading-zero strip, so it agrees with the engine at all output lengths:

- 64 bytes → `L = 512 = 0x0200` → engine `0x02 0x00` (2 bytes); this module
  `0x02 0x00`. Identical.
- 32 bytes → `L = 256 = 0x0100` → engine `0x01 0x00`; this module `0x01 0x00`.
  Identical.
- 16 bytes → `L = 128 = 0x80` → engine `0x80` (1 byte); this module `0x80`.
  Identical (the 1-byte branch, which the old monorepo wrapper could not reach).

The old monorepo wrapper always emitted 2 bytes (`binary.BigEndian.PutUint16`),
which agreed only for `L >= 256`. This module is more general.

### D3 — counter `[i]_b` width is `R`, big-endian, takes the LOW `R` bytes

Engine: `gost_keyexpimp.c:234-236` computes `iter_net = be32(i)` (4-byte
big-endian) and points `rep_ptr` at `((unsigned char*)&iter_net) + (4 -
representation)`, i.e. the **last `R` bytes** of the big-endian counter. For
`R = 1` that is the single least-significant byte.

This module (`kdftree/kdftree.go:91-97`, `counterBytes`) stores `i` as an
8-byte big-endian integer and returns the last `r` bytes — the low `r` bytes
of the big-endian counter, correctly handling `R = 1..4`. The old monorepo
wrapper hardcoded `R = 1` only. **For `R > 1` you must write the low `R`
bytes of `i` big-endian; do not zero-pad on the wrong side.**

### D4 — Streebog-256 internal endianness (inherited, not in this file)

The KDF input layout above is byte-oriented and endianness-neutral *at this
layer*. But the underlying Streebog-256 hash (GOST R 34.11-2012, RFC 6986)
is full of little-endian block handling internally. That is entirely
encapsulated inside `github.com/bigbes/gostcrypto/streebog`. Validate the
hash independently against RFC 6986 KATs **before** testing the KDF — a
wrong-endian Streebog will pass nothing here and the failure will look like a
KDF bug.

### D5 — no destructive-state hazard in this path

This module creates a **fresh** `hmac.New(...)` per iteration
(`kdftree/kdftree.go:76`) and calls `Sum(out)` once on each. There
is no reused/aliased state across iterations, no key-meshing, and no
deferred-last-block behavior. The three known gogost↔engine divergences in
`TODO.md` (S-box row order, R 34.11-94 empty-input finalization, CryptoPro
key meshing) **do not touch this primitive**: it never uses GOST 28147,
never uses R 34.11-94, and never hashes empty input (the HMAC message always
contains at least `i_b || 0x00 || L_b`).

### D6 — gogost-vs-module argument order

gogost's `Derive(dst, label, seed)` returns `h.Sum(dst)` (appends to `dst`).
This module's `KDFTree256(key, label, seed, r, outLen)` returns a fresh slice.
Different signature; do not assume the gogost ordering when porting.

## Test vectors

### KAT-1 — gost-engine etalon (R = 1, 64-byte output) — AUTHORITATIVE

Source: `tmp/engine/test_keyexpimp.c:78-97,164-165`. This is the parity
target (Tarantool's upstream). Inputs are all hex.

```
key   (32 B) = 000102030405060708090A0B0C0D0E0F
               101112131415161718191A1B1C1D1E1F
label (4 B)  = 26BDB878
seed  (8 B)  = AF21434145656378
R            = 1
keyOutLen    = 64 bytes   (L = 512 bits  -> L_b = 0x02 0x00)
```

Expected output (64 bytes, `kdf_etalon`, `test_keyexpimp.c:88-97`):

```
22B6837845C6BEF65EA71672B2658310
86D3C76AEBE6DAE91CAD51D83F79D16B
074C9330599D7F8D712FCA54392F4DDD
E93751206B3584C8F43F9E6DC51531F9
```

Trace of the two HMAC iterations (both use Streebog-256, key = `key`):

```
iter 1 message = 01 26BDB878 00 AF21434145656378 0200
  -> HMAC = 22B6837845C6BEF65EA71672B265831086D3C76AEBE6DAE91CAD51D83F79D16B
iter 2 message = 02 26BDB878 00 AF21434145656378 0200
  -> HMAC = 074C9330599D7F8D712FCA54392F4DDDE93751206B3584C8F43F9E6DC51531F9
out = iter1 || iter2
```

Note the counter byte (`01` then `02`) is the only difference between the
two messages; `L_b = 0200` is identical in both.

### KAT-2 — single-block (R = 1, 32-byte output) — RFC 7836 Appendix B example 9

Source: `rfc7836.txt:1499-1526`. Same key/label/seed as KAT-1; `L = 256`,
so `[L]_b = 0x01 0x00`. The HMAC message is:

```
01 26BDB878 00 AF21434145656378 0100
```

Expected output (32 bytes):

```
a1aa5f7de402d7b3d323f2991c8d4534013137010a83754fd0af6d7cd4922ed9
```

This is also the single-block `KDF_GOSTR3411_2012_256` (RFC 7836 §4.5), which
equals `KDF_TREE` with `R=1, L=256`. It is an authoritative RFC vector.

Note: `KDFTree256(key, label, seed, 1, 32)` uses `L_b = 0x0100`, which differs
from `KDFTree256(key, label, seed, 1, 64)` which uses `L_b = 0x0200`. The two
outputs are entirely different (not a prefix truncation).

## Re-implementation checklist

Each step is independently testable against a vector.

1. **Streebog-256 hash.** Use the clean-room implementation in
   `github.com/bigbes/gostcrypto/streebog`. Verify against RFC 6986 KATs.
   Output 32 bytes, block 64 bytes. (D4)
2. **HMAC-Streebog-256.** Wrap with `crypto/hmac`. Verify the HMAC block size
   is 64 bytes and output is 32 bytes. A wrong block size silently corrupts
   the `opad`/`ipad` padding.
3. **Counter encoding.** Implement `[i]_b` as the low `R` bytes of `i`,
   big-endian (`R = 1` → single byte). Cross-check against engine
   `gost_keyexpimp.c:234-236`. (D3)
4. **Length encoding.** Compute `L = outLen * 8`. Encode `[L]_b`
   big-endian with leading zeros stripped
   (engine `gost_keyexpimp.c:227-231`). (D2)
5. **Message assembly.** For `i = 1 .. ceil(outLen/32)`, build
   `[i]_b || label || 0x00 || seed || [L]_b` and HMAC it with `key`. Note the
   literal `0x00` separator and that `[L]_b` is identical across iterations.
6. **Concatenate.** Append each 32-byte HMAC block; truncate to `outLen` bytes.
   Validate: `r` in 1..4, `outLen > 0`, not exceeding the r-byte counter range.
7. **Validate against KAT-1** (64-byte RFC 7836 example 10 / engine etalon).
   Both blocks must match bit-for-bit.
8. **Validate against KAT-2** (32-byte RFC 7836 example 9). Single byte must
   match `a1aa5f7de402d7b3...`.
