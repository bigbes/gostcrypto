# KDF_TREE_GOSTR3411_2012_256 (KDFTree2012_256)

## What it is

`KDFTree2012_256` is a counter-mode key-derivation function built on
HMAC with the Streebog-256 hash (GOST R 34.11-2012, 256-bit output). It
takes a derivation key plus a `label` and a `seed` and expands them into
an arbitrary multiple of 32 bytes of keying material. It is the KDF defined
in **RFC 7836 §4.4** ("The Key Derivation Function
KDF_TREE_GOSTR3411_2012_256"), also published as Russian recommendation
**R 50.1.113-2016 §4.5**. The single-block variant (RFC 7836 §4.5,
`KDF_GOSTR3411_2012_256`) is just this function with `R = 1` and `L = 256`,
i.e. exactly the first (and only) iteration producing 32 bytes.

*Intended implementer: a Sonnet-class coding agent — every constant, S-box, parameter table, and edge case is inlined so this primitive can be built without consulting gogost or external specs.*

In gost-engine the C entry point is `gost_kdftree2012_256()`
(`tmp/engine/gost_keyexpimp.c:201`). It is a workhorse used by three
distinct higher-level constructions in Tarantool's GOST stack:

- **KEG / key expansion** for RFC 9367 ciphers — `internal/gost/keg_gost.go:77`
  calls `KDFTree2012_256(tmpkey, "kdf tree", ukmSource[16:24], 64)` to split a
  VKO-derived `tmpkey` into a 64-byte `(cipher_key || mac_key)` pair. The
  matching engine path is `gost_gost2015.c:173`
  (`gost_kdftree2012_256(keys, 64, inkey, 32, ...)`).
- **TLSTREE** key meshing — the engine's `gost_tlstree()`
  (`tmp/engine/gost_keyexpimp.c:261`) chains three `gost_kdftree2012_256`
  calls with labels `"level1"`, `"level2"`, `"level3"` and masked TLS
  sequence numbers as seeds. (Our pure-Go TLS path uses gogost's `TLSTree`
  wrapper for this, but the underlying primitive is the same KDF tree.)
- **kexp15 / VKO key transport** — `gost_ec_keyx.c:166`.

### Repository status

`statusKind = gogost-backed`. The repo's wrapper
`internal/gost/kdftree_gost.go` does **not** call gogost's `KDF.Derive`;
instead it constructs `crypto/hmac` over `gost34112012256.New`
(gogost's Streebog-256 `hash.Hash` constructor) and drives the iteration
loop itself. So the only gogost dependency for this primitive is the
**Streebog-256 hash function** (`go.stargrave.org/gogost/v7/gost34112012256.New`,
imported at `internal/gost/kdftree_gost.go:18`). A GPL-free reimplementation
of this file needs only a license-clean HMAC-Streebog-256; the counter loop
below is original.

The reason the wrapper re-rolls the loop rather than calling gogost's
`KDF.Derive`: gogost's `Derive`
(`third_party/gogost/gost34112012256/kdf.go:31`) **hardcodes** the iteration
counter to the single byte `0x01` (line 32) and the length suffix to the two
bytes `0x01 0x00` (lines 44-47). It therefore only ever emits 32 bytes
(`L = 256` bits) in one HMAC call and cannot produce the 64-byte output the
KEG step needs. See "RFC ↔ implementation deltas" below.

## Specification

### HMAC core (RFC 7836 §4.1.1)

`HMAC_GOSTR3411_2012_256` is plain HMAC (RFC 2104) instantiated with:

- Hash `H_256` = GOST R 34.11-2012 with 256-bit output (Streebog-256, RFC 6986).
- HMAC output size: **32 bytes**.
- HMAC block size (the `ipad`/`opad` width): **64 bytes**.

`ipad`/`opad` are the standard RFC 2104 constants (`0x36`/`0x5c` repeated).
Go's `crypto/hmac` handles all of this given a `func() hash.Hash` for
Streebog-256.

### KDF_TREE (RFC 7836 §4.4)

Inputs:

| Symbol  | Meaning                                                            |
|---------|-------------------------------------------------------------------|
| `K_in`  | derivation key (HMAC key); any length                             |
| `label` | protocol-assigned byte string, fixed length per use site          |
| `seed`  | protocol-assigned byte string, fixed length per use site          |
| `R`     | width in bytes of the iteration-counter field (1, 2, 3, or 4)     |
| `L`     | required output size **in bits**, `L <= 256 * (2^(8*R) - 1)`      |

Output, normative formula (RFC 7836 §4.4):

```
KDF_TREE_GOSTR3411_2012_256(K_in, label, seed, R) = K(1) | K(2) | K(3) | ...

K(i) = HMAC_GOSTR3411_2012_256(K_in, [i]_b | label | 0x00 | seed | [L]_b)
```

where:

- `[i]_b` — byte representation of the iteration counter `i`, **network byte
  order (big-endian)**, exactly `R` bytes wide.
- `0x00` — a single literal zero separator byte between `label` and `seed`.
- `[L]_b` — byte representation of `L` (the bit-length of the *total* desired
  output), **network byte order (big-endian)**, **variable length with no
  leading zero bytes added** (RFC 7836 §4.4: "no leading zero bytes added").
- `|` denotes concatenation.

The number of iterations is `ceil(L / 256)`. Each `K(i)` is a full 32-byte
HMAC block; the concatenation is truncated to `L` bits if `L` is not a
multiple of 256. (Our wrapper restricts `L` to multiples of 256 bits, so no
truncation occurs — see deltas.)

### Single-block KDF (RFC 7836 §4.5)

```
KDF_GOSTR3411_2012_256(K_in, label, seed)
  = HMAC_GOSTR3411_2012_256(K_in, 0x01 | label | 0x00 | seed | 0x01 | 0x00)
```

This is `KDF_TREE(..., R=1)` with `L = 256`: `[1]_b = 0x01`,
`[256]_b = 0x01 0x00`. Exactly what gogost's `KDF.Derive` hardcodes.

### Parameters used by this repo

The repo always calls with:

- `R = 1` (1-byte counter; the `representation` argument is `1` at every
  engine call site — `gost_keyexpimp.c:296-301`, `gost_gost2015.c:173`).
- `keyOutLen` a positive multiple of **32 bytes** (the wrapper panics
  otherwise — `internal/gost/kdftree_gost.go:28`). For KEG it is 64.
- `L = keyOutLen * 8`. For 64 bytes, `L = 512`, and `[L]_b = 0x02 0x00`.

Because `R = 1`, the counter field is a single byte `0x01, 0x02, ...`. With
`keyOutLen <= 32*255 = 8160` bytes the 1-byte counter never overflows; the
wrapper additionally caps `keyOutLen <= 0xFFFF/8 = 8191` so `L` fits the
naive 2-byte length encoding (`internal/gost/kdftree_gost.go:31`).

### Per-iteration HMAC input layout (R = 1)

```
  +------+----------------+------+---------------+----------------+
  | i_b  |     label      | 0x00 |     seed      |     L_b        |
  | 1 B  |  len(label) B  | 1 B  |  len(seed) B  | 1 or 2 B (BE)  |
  +------+----------------+------+---------------+----------------+
   i = 1..ceil(L/256), big-endian
```

`L_b` is the **same value for every iteration** (it encodes the total output
length, not the per-block length). Only `i_b` changes between iterations.

## RFC ↔ implementation deltas

### D1 — gogost `KDF.Derive` is the §4.5 single-block KDF only, NOT the tree

`third_party/gogost/gost34112012256/kdf.go:31-53` implements exactly
`KDF_GOSTR3411_2012_256` (RFC 7836 §4.5): counter byte hardcoded to `0x01`
(line 32) and length suffix hardcoded to `0x01 0x00` (lines 44-47, i.e.
`L = 256`). It produces **32 bytes, once**. It is *not* a multi-iteration
tree and cannot derive 64 bytes. The repo wrapper therefore ignores
`KDF.Derive` and re-rolls the loop
(`internal/gost/kdftree_gost.go:40-48`), changing the counter byte per
iteration and emitting `L_b = 0x02 0x00` for 64-byte output. **A
reimplementer who naively wires up gogost's `Derive` will get the wrong
answer for any output longer than 32 bytes.** (Documented in
`CLAUDE.md` → "gogost/v7 library gotchas" → `KDF.Derive`.)

### D2 — `[L]_b` strips leading zero bytes (engine), wrapper does NOT (but they agree for our sizes)

RFC 7836 §4.4 says `[L]_b` carries **no leading zero bytes**. The engine
enforces this literally: `gost_keyexpimp.c:212` computes
`len_repr = be32(keyout_len * 8)` as a 4-byte big-endian integer, then
`gost_keyexpimp.c:227-231` walks `len_ptr` forward while the leading byte is
zero, shrinking `len_repr_len` from 4 down to the minimal width.

The repo wrapper instead **always emits 2 bytes** big-endian
(`internal/gost/kdftree_gost.go:36-37,46`: `binary.BigEndian.PutUint16`).

These agree **only because** every value the repo uses has its high byte
non-zero in the low 2 bytes and its top 2 bytes zero:

- 64 bytes → `L = 512 = 0x0200` → engine strips to `0x02 0x00` (2 bytes);
  wrapper emits `0x02 0x00`. Identical.
- 32 bytes → `L = 256 = 0x0100` → engine `0x01 0x00`; wrapper `0x01 0x00`.
  Identical.

The divergence would surface only for `L < 256` bits (engine: 1 byte;
wrapper: 2 bytes) — impossible here, since `keyOutLen` is a multiple of 32
bytes so `L >= 256`. **If you reimplement for general `L`, replicate the
leading-zero strip from `gost_keyexpimp.c:227-231`, otherwise you will
mismatch the engine for sub-256-bit or >65535-bit outputs.**

### D3 — counter `[i]_b` width is `R`, big-endian, takes the LOW `R` bytes

Engine: `gost_keyexpimp.c:234-236` computes `iter_net = be32(i)` (4-byte
big-endian) and points `rep_ptr` at `((unsigned char*)&iter_net) + (4 -
representation)`, i.e. the **last `R` bytes** of the big-endian counter. For
`R = 1` that is the single least-significant byte. The repo wrapper hardcodes
this to one byte: `h.Write([]byte{byte(i)})`
(`internal/gost/kdftree_gost.go:42`). Correct for `R = 1` only. **For
`R > 1` you must write the low `R` bytes of `i` big-endian; do not zero-pad
on the wrong side.**

### D4 — Streebog-256 internal endianness (inherited, not in this file)

The KDF input layout above is byte-oriented and endianness-neutral *at this
layer*. But the underlying Streebog-256 hash (GOST R 34.11-2012, RFC 6986)
is full of little-endian block handling internally. That is entirely
encapsulated inside gogost's `gost34112012256.New`. If you replace the hash
with your own Streebog-256, validate the hash independently against RFC 6986
KATs **before** testing the KDF — a wrong-endian Streebog will pass nothing
here and the failure will look like a KDF bug.

### D5 — no destructive-state hazard in this path

Unlike gogost's `gost28147.MAC.Sum` (CLAUDE.md gotcha), the KDF wrapper
creates a **fresh** `hmac.New(...)` per iteration
(`internal/gost/kdftree_gost.go:41`) and calls `Sum(nil)` once on each. There
is no reused/aliased state across iterations, no key-meshing, and no
deferred-last-block behavior. The three known gogost↔engine divergences in
`TODO.md` (S-box row order, R 34.11-94 empty-input finalization, CryptoPro
key meshing) **do not touch this primitive**: it never uses GOST 28147,
never uses R 34.11-94, and never hashes empty input (the HMAC message always
contains at least `i_b || 0x00 || L_b`).

### D6 — gogost-vs-repo argument order

gogost's `Derive(dst, label, seed)` returns `h.Sum(dst)` (appends to `dst`).
The repo wrapper takes `(key, label, seed, keyOutLen)` and returns a fresh
slice. Different signature; do not assume the gogost ordering when porting.

## Test vectors

### KAT-1 — gost-engine etalon (R = 1, 64-byte output) — AUTHORITATIVE

Source: `tmp/engine/test_keyexpimp.c:78-97,164-165`. This is the parity
target (Tarantool's upstream). Inputs are all hex.

```
key   (32 B) = 000102030405060708090A0B0C0D0E0F
               101112131415161718191A1B1C1D1E1F
label (4 B)  = 26BDB878
seed  (8 B)  = AF21434145656378
R            = 1
keyOutLen    = 64 bytes   (L = 512 bits  -> L_b = 0x02 0x00)
```

Expected output (64 bytes, `kdf_etalon`, `test_keyexpimp.c:88-97`):

```
22B6837845C6BEF65EA71672B2658310
86D3C76AEBE6DAE91CAD51D83F79D16B
074C9330599D7F8D712FCA54392F4DDD
E93751206B3584C8F43F9E6DC51531F9
```

Trace of the two HMAC iterations (both use Streebog-256, key = `key`):

```
iter 1 message = 01 26BDB878 00 AF21434145656378 0200
  -> HMAC = 22B6837845C6BEF65EA71672B265831086D3C76AEBE6DAE91CAD51D83F79D16B
iter 2 message = 02 26BDB878 00 AF21434145656378 0200
  -> HMAC = 074C9330599D7F8D712FCA54392F4DDDE93751206B3584C8F43F9E6DC51531F9
out = iter1 || iter2
```

Note the counter byte (`01` then `02`) is the only difference between the
two messages; `L_b = 0200` is identical in both.

### KAT-2 — single-block (R = 1, 32-byte output) — derived check

Take the first 32 bytes of KAT-1's input layout with `L_b = 0x01 0x00`
(i.e. `keyOutLen = 32`, `L = 256`). This is exactly gogost's `KDF.Derive`
shape (counter `0x01`, suffix `0x01 0x00`). Use it to cross-check a
license-clean HMAC-Streebog-256 against gogost's `KDF.Derive(nil, label, seed)`
with the same key — they must agree byte-for-byte on 32 bytes. (No
standalone engine etalon for the 32-byte case is checked in; this is a
consistency vector, not an authoritative one.)

### Existing repo tests

There is currently **no dedicated `kdftree_*_test.go`**. Coverage is
indirect, through the KEG path (`internal/gost/keg_gost_test.go`) which calls
`KDFTree2012_256(tmpkey, "kdf tree", ukmSource[16:24], 64)`
(`internal/gost/keg_gost.go:77`) and validates the full key-expansion output
against engine vectors. **Recommendation for the reimplementer:** add a
direct KAT test using KAT-1 above before touching the KEG layer, so a KDF
regression is not masked by the surrounding KEG/VKO machinery.

## Re-implementation checklist

Each step is independently testable against a vector.

1. **Streebog-256 hash.** Implement (or vendor, license-clean) GOST R
   34.11-2012 256-bit. Verify against RFC 6986 KATs first. Output 32 bytes,
   block 64 bytes. (D4)
2. **HMAC-Streebog-256.** Wrap step 1 in RFC 2104 HMAC. Verify the HMAC
   block size is 64 bytes and output is 32 bytes. A wrong block size silently
   corrupts the `opad`/`ipad` padding.
3. **Counter encoding.** Implement `[i]_b` as the low `R` bytes of `i`,
   big-endian (`R = 1` → single byte). Cross-check against engine
   `gost_keyexpimp.c:234-236`. (D3)
4. **Length encoding.** Compute `L = keyOutLen * 8`. Encode `[L]_b`
   big-endian. For full generality, strip leading zero bytes
   (engine `gost_keyexpimp.c:227-231`); for the repo's 32-byte-multiple sizes,
   a fixed 2-byte big-endian encoding is equivalent. (D2)
5. **Message assembly.** For `i = 1 .. keyOutLen/32`, build
   `[i]_b || label || 0x00 || seed || [L]_b` and HMAC it with `key`. Note the
   literal `0x00` separator and that `[L]_b` is identical across iterations.
6. **Concatenate.** Append each 32-byte HMAC block; the result is exactly
   `keyOutLen` bytes (no truncation when `keyOutLen` is a multiple of 32).
   Reject `keyOutLen <= 0`, non-multiples of 32, and oversized requests
   (mirror `internal/gost/kdftree_gost.go:28-33`).
7. **Validate against KAT-1** (64-byte engine etalon). Both blocks must match
   bit-for-bit.
8. **Validate the KEG integration** (`internal/gost/keg_gost_test.go`) to
   confirm the KDF behaves correctly inside the larger construction.

## Conformance & fuzz testing

Drop the scaffolding below into your clean-room package to prove parity. The
strategy is **differential testing** against two references: (1)
`internal/gost.KDFTree2012_256` — the local, corrected 64-byte iterator that is
the de-facto spec this repo matches (`internal/gost/kdftree_gost.go:27`); and
(2) the pinned RFC 7836 / gost-engine KAT-1 vector above, which anchors the
local impl to an authoritative external etalon. Raw gogost
`gost34112012256.NewKDF(key).Derive(dst, label, seed)` is **only usable for the
32-byte case** — it hardcodes the counter byte `0x01` and the length suffix
`0x01 0x00` (`third_party/gogost/gost34112012256/kdf.go:31-53`, see D1), so it
physically cannot emit 64 bytes; diff against it *only* at `keyOutLen == 32`.
Fuzz random `key`/`label`/`seed` at output lengths **32 and 64**: at 32 all
three references must agree, at 64 your impl must match `internal/gost` (and the
pinned vector when the fuzzed inputs happen to equal KAT-1's).

### KAT — pinned vectors (`*_test.go`, `//go:build gost`)

Seeded with the exact KAT-1 / KAT-2 bytes from this doc — do not invent new ones.

```go
//go:build gost

package yourpkg

import (
	"bytes"
	"encoding/hex"
	"testing"

	gostref "go.bigb.es/tlsdialer/internal/gost"
	mynew "go.bigb.es/tlsdialer/yourpkg" // clean-room impl under test
	"go.stargrave.org/gogost/v7/gost34112012256"
)

func mustHex(t *testing.T, s string) []byte {
	t.Helper()
	b, err := hex.DecodeString(s)
	if err != nil {
		t.Fatalf("bad hex %q: %v", s, err)
	}
	return b
}

func TestKDFTree2012_256Conformance(t *testing.T) {
	// KAT-1: gost-engine etalon, R=1, 64-byte output (test_keyexpimp.c:78-97).
	key := mustHex(t, "000102030405060708090A0B0C0D0E0F101112131415161718191A1B1C1D1E1F")
	cases := []struct {
		name      string
		label     []byte
		seed      []byte
		keyOutLen int
		want      string // "" => no pinned etalon, cross-check refs only
	}{
		{
			name:      "KAT-1/64B",
			label:     mustHex(t, "26BDB878"),
			seed:      mustHex(t, "AF21434145656378"),
			keyOutLen: 64,
			want: "22B6837845C6BEF65EA71672B265831086D3C76AEBE6DAE91CAD51D83F79D16B" +
				"074C9330599D7F8D712FCA54392F4DDDE93751206B3584C8F43F9E6DC51531F9",
		},
		{
			// KAT-2: same inputs truncated to 32B (L_b = 0x01 0x00); gogost-comparable.
			name:      "KAT-2/32B",
			label:     mustHex(t, "26BDB878"),
			seed:      mustHex(t, "AF21434145656378"),
			keyOutLen: 32,
			want:      "", // consistency vector only, no checked-in engine etalon
		},
	}

	for _, tc := range cases {
		t.Run(tc.name, func(t *testing.T) {
			got := mynew.KDFTree2012_256(key, tc.label, tc.seed, tc.keyOutLen)
			if len(got) != tc.keyOutLen {
				t.Fatalf("len = %d, want %d", len(got), tc.keyOutLen)
			}
			// Reference 1: in-repo corrected iterator.
			if ref := gostref.KDFTree2012_256(key, tc.label, tc.seed, tc.keyOutLen); !bytes.Equal(got, ref) {
				t.Fatalf("mismatch vs internal/gost:\n got %x\n ref %x", got, ref)
			}
			// Reference 2: pinned authoritative etalon, when present.
			if tc.want != "" {
				if want := mustHex(t, tc.want); !bytes.Equal(got, want) {
					t.Fatalf("mismatch vs pinned vector:\n got  %x\n want %x", got, want)
				}
			}
			// Reference 3: raw gogost — ONLY valid for the 32-byte single-block case (D1).
			if tc.keyOutLen == 32 {
				ref := gost34112012256.NewKDF(key).Derive(nil, tc.label, tc.seed)
				if !bytes.Equal(got, ref) {
					t.Fatalf("mismatch vs gogost KDF.Derive:\n got %x\n ref %x", got, ref)
				}
			}
		})
	}
}
```

### Fuzz harness (`*_test.go`, `//go:build gost`)

Seeds from the KAT inputs, then normalizes the random `[]byte` into this
primitive's fixed-size arguments (`key` is padded/truncated to 32 bytes; `label`
and `seed` are free-length; `keyOutLen` is forced to a legal {32, 64}). It runs
the clean-room impl and the references on identical inputs and fails on any
divergence. This is the randomized-primitive form — there is no OID table here
(that pattern belongs to the curve primitives), so the fuzzer just diffs raw
outputs.

```go
//go:build gost

package yourpkg

import (
	"bytes"
	"testing"

	gostref "go.bigb.es/tlsdialer/internal/gost"
	mynew "go.bigb.es/tlsdialer/yourpkg"
	"go.stargrave.org/gogost/v7/gost34112012256"
)

func FuzzKDFTree2012_256Conformance(f *testing.F) {
	// Seed corpus from the KAT inputs above.
	f.Add([]byte("\x00\x01\x02\x03\x04\x05\x06\x07\x08\x09\x0a\x0b\x0c\x0d\x0e\x0f"+
		"\x10\x11\x12\x13\x14\x15\x16\x17\x18\x19\x1a\x1b\x1c\x1d\x1e\x1f"),
		[]byte("\x26\xbd\xb8\x78"), []byte("\xaf\x21\x43\x41\x45\x65\x63\x78"), uint8(64))
	f.Add([]byte("short-key"), []byte("level1"), []byte("\x00\x00\x00\x00\x00\x00\x00\x01"), uint8(32))

	f.Fuzz(func(t *testing.T, rawKey, label, seed []byte, lenSel uint8) {
		// Normalize: HMAC key fixed to 32 bytes; output length to a legal {32,64}.
		key := make([]byte, 32)
		copy(key, rawKey)
		keyOutLen := 32
		if lenSel&1 == 1 {
			keyOutLen = 64
		}

		got := mynew.KDFTree2012_256(key, label, seed, keyOutLen)
		if len(got) != keyOutLen {
			t.Fatalf("len = %d, want %d", len(got), keyOutLen)
		}

		// Reference 1: in-repo iterator — must agree at every legal length.
		if ref := gostref.KDFTree2012_256(key, label, seed, keyOutLen); !bytes.Equal(got, ref) {
			t.Fatalf("mismatch vs internal/gost (len=%d):\n got %x\n ref %x", keyOutLen, got, ref)
		}

		// Reference 2: raw gogost — single-block only (D1: 32B, hardcoded suffix).
		if keyOutLen == 32 {
			ref := gost34112012256.NewKDF(key).Derive(nil, label, seed)
			if !bytes.Equal(got, ref) {
				t.Fatalf("mismatch vs gogost KDF.Derive:\n got %x\n ref %x", got, ref)
			}
		}
	})
}
```

> Note: this primitive has a pure-Go reference (`internal/gost` + gogost), so no
> CLI oracle is needed. Primitives whose only reference is the gost-engine CLI
> (OMAC, CTR-ACPKM, KEG, KExp15, KeyWrap) would instead shell out to the oracle
> from `CLAUDE.md` (e.g. `openssl dgst -engine gost -mac kuznyechik-mac ...`) via
> a small `exec.Command` helper in place of the `gostref` import — KDFTree does
> not.
