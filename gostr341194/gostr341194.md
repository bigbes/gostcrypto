# GOST R 34.11-94 legacy hash (CryptoPro param set)

`GOST R 34.11-94` is the legacy Soviet/Russian 256-bit hash function, superseded
by Streebog (GOST R 34.11-2012). It is built on top of the `GOST 28147-89` block
cipher: the compression function keys four ECB encryptions from the chaining
value and message block, then runs a linear mixing transform. This document
covers the **CryptoPro parameter set** variant — the one used on the wire by
Tarantool-EE — which selects a specific S-box (`id-GostR3411-94-CryptoProParamSet`,
OID `1.2.643.2.2.30.1`, RFC 4357 §11.2 / §10.4) for the inner 28147 cipher.

*Intended implementer: a Sonnet-class coding agent — every constant, S-box, parameter table, and edge case is inlined so this primitive can be built without consulting gogost or external specs.*

- GOST identity: **GOST R 34.11-94**, RFC 5831.
- Hash algorithm OID: **id-GostR3411-94 = 1.2.643.2.2.9** (RFC 4357 §10.4;
  mirrored in this repo at `x509gost/oids.go:49`).
- Parameter-set OID: **id-GostR3411-94-CryptoProParamSet = 1.2.643.2.2.30.1**
  (RFC 4357 §10.4 / §11.2).
- Output / block / internal state: **256 bits (32 bytes)** each.

### Where this repo uses it

This primitive is a **clean-room BSD implementation** in `gostr341194/`.  The
module carries zero GPL code — it does not wrap or depend on gogost anywhere
(see `CLAUDE.md` "Clean-room constraint"). Consumers in this module:

- Root facade `exports.go` — `NewGOSTR341194CryptoProHash()` returns a

In-package tests live in `gostr341194/gostr341194_test.go`. Differential

## Specification

All step references are to **RFC 5831**. GOST R 34.11-94 is little-endian and
"reads words right to left" (RFC 5831 §3 byte-ordering note): the rightmost
byte of a 256-bit word is numbered 1. Implementations that work on byte arrays
must reverse blocks at the boundaries (see deltas).

### Sizes and constants

| Item | Value |
| --- | --- |
| Block size | 32 bytes (256 bits) |
| Output size | 32 bytes (256 bits) |
| Internal state `H` | 32 bytes, initialized to all-zero (`h0 = 0^256`) |
| Checksum `Σ` (SIGMA) | 32 bytes, addition modulo 2^256, init 0 |
| Length `L` | 32 bytes, total message bit-length mod 2^256, init 0 |
| Inner cipher | GOST 28147-89, ECB, 32-round encrypt |

Compression-function constants `C2`, `C3`, `C4` (RFC 5831 §5.1; gogost
`third_party/gogost/gost341194/hash.go:36-53`). `C2` and `C4` are all-zero.
`C3` (32 bytes, in the byte order gogost uses) is:

```
ff 00 ff ff 00 00 00 ff ff 00 00 ff 00 ff ff 00
00 ff 00 ff 00 ff 00 ff ff 00 ff 00 ff 00 ff 00
```

RFC 5831 §5.1 specifies C3 as the bit pattern
`1^8 0^8 1^16 0^24 1^16 0^8 (0^8 1^8)^2 1^8 0^8 (0^8 1^8)^4 (1^8 0^8)^4`.

### CryptoPro S-box (the parameter dependency)

The compression function's inner 28147 cipher uses the **CryptoPro hash
S-box**, 8 rows × 16 nibbles (RFC 4357 §11.2; gogost
`third_party/gogost/gost28147/sbox.go:93-102`,
`SboxIdGostR341194CryptoProParamSet`):

```
row0: 10  4  5  6  8  1  3  7 13 12 14  0  9  2 11 15
row1:  5 15  4  0  2 13 11  9  1  7  6  3 12 14 10  8
row2:  7 15 12 14  9  4  1  0  3 11  5  2  6 10  8 13
row3:  4 10  7 12  0 15  2  8 14  1  6  5 13 11  9  3
row4:  7  6  4 11  9 12  2 10  1  8  0 14 15 13  3  5
row5:  7  6  2  4 13  9 15  0 10  1  5 11  8 14 12  3
row6: 13 14  4  1  7  0  5 10  3 12  8 15  6  2  9 11
row7:  1  3 10  9  5 11  4 15  8  6  7 14 13  0  2 12
```

Swapping this for `SboxIdGostR341194TestParamSet` (gogost `SboxDefault`,
hash.go:34) gives the RFC-5831 *test* vectors instead; the CryptoPro set is the
one Tarantool uses.

### Compression function χ(M, H) (RFC 5831 §5)

Inputs `H, M ∈ V256`. Three stages:

**1. Key generation (§5.1).** Produce four 256-bit keys `K1..K4` (each a
GOST 28147-89 key) via transforms `A` and `P`:

- `A(X)` (RFC 5831 §5.1): split `X = x4||x3||x2||x1` into four 64-bit words;
  `A(X) = (x1 ⊕ x2) || x4 || x3 || x2`. gogost: `fA`, hash.go:93-105.
- `P(X)` (§5.1): the byte permutation `φ(i+1+4(k-1)) = 8i+k`, i=0..3, k=1..8.
  gogost: `fP`, hash.go:107-115.

Algorithm:
```
U := H ; V := M ; W := U ⊕ V ; K1 := P(W)
U := A(U) ⊕ C2 ; V := A(A(V)) ; W := U ⊕ V ; K2 := P(W)
U := A(U) ⊕ C3 ; V := A(A(V)) ; W := U ⊕ V ; K3 := P(W)
U := A(U) ⊕ C4 ; V := A(A(V)) ; W := U ⊕ V ; K4 := P(W)
```
(gogost interleaves key gen with encryption; `step`, hash.go:135-211.)

**2. Encryption (§5.2).** Split `H = h4||h3||h2||h1` (four 64-bit words).
`si = E(Ki, hi)` (GOST 28147-89 ECB 32-round encrypt). `S = s4||s3||s2||s1`.

**3. Mixing (§5.3).** Define ψ on `η16||...||η1` (16-bit words):
`ψ(X) = (η1⊕η2⊕η3⊕η4⊕η13⊕η16) || η16 || η15 || ... || η2`.
gogost: `fChi`, hash.go:117-123. Final output:

```
χ(M, H) = ψ^61( H ⊕ ψ( M ⊕ ψ^12(S) ) )
```
i.e. 12 ψ over S, XOR M, 1 ψ, XOR H, then 61 ψ — 74 total ψ applications
(gogost hash.go:213-221; engine `transform_3` loop `gosthash.c:138-144`).

### Main procedure (RFC 5831 §6)

Initialize `H := 0^256`, `Σ := 0^256`, `L := 0^256`. For each full 256-bit
message block `M_s` (processed in order):

```
H := χ(M_s, H)
L := (L + 256) mod 2^256
Σ := Σ ⊕'  M_s        # ⊕' = byte-wise addition mod 2^256 (with carry)
```

Final (partial or last) block of `|M|` bits, with `M' = 0^(256-|M|) || M`
(left zero-pad of the last fragment):

```
L := (L + |M|) mod 2^256
Σ := Σ ⊕' M'
H := χ(M', H)
H := χ(L, H)     # absorb the length block
H := χ(Σ, H)     # absorb the checksum block
```

The 256-bit `H` is the digest. Note `Σ` uses **addition mod 2^256**
(`add_blocks` with carry, `gosthash.c:64-74`), NOT XOR — RFC 5831 writes it
as the `(+)'` operator (§6).

## RFC ↔ implementation deltas

THE CORE SECTION. Every place gogost / gost-engine reinterpret or under-specify
the RFC. Each cites the RFC and the source line.

### D1. Empty-input finalization — gogost ≠ gost-engine (KNOWN DIVERGENCE)

For an empty message, **the two implementations disagree** and only one matches
Tarantool's upstream engine. This is divergence #2 in `TODO.md`
("Disagreements", TODO.md:7-10).

- **gost-engine** (`tmp/engine/gosthash.c:256-258`): in `finish_hash`, after
  the partial-block path, when `fin_len == 0` it runs an **extra**
  `hash_step(c, H, zero_block)`. So an empty input gets **three** compression
  calls total: this extra zero-block step, then the length block, then the
  checksum block. Result for CryptoPro empty input:
  `3f25bc1fbbce27ca10fb1958f319473ae7e17482c3b53ecf47a7e2de8aabe4c8`
  (`tmp/engine/tcl_tests/dgst.try:89`).
- **gogost** (`third_party/gogost/gost341194/hash.go:246-267`): `Sum` only
  runs the length block (`step`, hash.go:260) and the checksum block
  (hash.go:264) — **two** compression calls for empty input, with no extra
  zero-block step. Result for CryptoPro empty input:
  `981e5f3ca30c841487830f84fb433e13ac1101569b9c13584ac483234cd656c0`
  (gogost test `TestVectorsCryptoProEmpty`, hash_test.go:151-160).

The RFC (§6) is silent on the degenerate empty case: its loop processes the
"last block" only when there is a message. The engine's extra step is the
de-facto correct behavior for Tarantool interop. A reimplementer MUST replicate
the engine's `fin_len == 0 → extra hash_step(H, 0^256)` to match Tarantool.

**Non-empty inputs agree bit-for-bit** between gogost and engine (TODO.md:9,
verified up to a 2100-byte vector `42e462ce...`). The divergence is *only*
empty input. In this repo it does not bite because the only TLS consumer is
HMAC-GOSTR341194 (PRF), which never hashes empty input
(`gost_suites.go:146`, TODO.md:77).

### D2. S-box row order is reversed inside gogost (compensated)

gogost stores `gost28147` S-box rows in **reverse order** and applies a
compensating `blockReverse` inside `step` before keying the cipher
(hash.go:144,163,181,199). The net cipher output **agrees** with engine's
`gost_enc_with_key` (TODO.md:9). A from-scratch implementation that uses the
S-box table exactly as written in RFC 4357 §11.2 / `sbox.go:93-102` (rows in
natural order) and a straightforward 28147 must NOT also reverse — only one of
{reverse rows, reverse block} is correct. Cross-check against the
non-empty KATs below before trusting your wiring. (See also `CLAUDE.md`
"S-box row order is a known gogost/engine divergence".)

### D3. Block-level little-endian reversal at every boundary

GOST words are little-endian (RFC 5831 §3). gogost reverses each 32-byte
message block on `Write` (`blockReverse(h.tmp, h.buf)`, hash.go:238) and on the
final partial block (hash.go:254), and reverses the digest on the way out
(hash.go:265). The engine instead keeps natural byte order throughout and bakes
the endianness into `swap_bytes`/`circle_xor8` and the per-word `gost_enc`. A
reimplementer must pick ONE convention and apply it consistently; mixing them
yields a byte-swapped digest. Easiest: follow gogost — reverse each input
32-byte block, reverse the output block, and reverse the 8-byte word fed to the
cipher within `step` (hash.go:147-149,165-167,183-185,201-203).

> **Note (the reversals are a self-canceling gogost artifact).** D3 and D4 are
> NOT a requirement of the algorithm — they describe gogost's *internal* choice
> to store every block big-endian (array index 0 = byte #32) and to compensate
> by reversing at the boundaries and inside `step`. A from-scratch implementation
> that instead reads RFC 5831's right-to-left numbering literally — array index
> `i` holds byte #(i+1), i.e. natural message order — should reverse **nothing**:
> no input-block reversal, no digest reversal, no per-word reversal in the inner
> cipher. The transforms `A`, `P`, `ψ`, the `step` word-feeding, the `Σ`/`L`
> blocks, and the C3 constant are then all written directly on that LSB-first
> numbering, and the output matches gogost/gost-engine bit-for-bit on every
> non-empty input. (Verified: the clean-room
> `gostr341194/` does exactly this and passes the differential test
> against the gogost oracle over 200+ random and boundary-length messages.) The
> rule "pick ONE convention and apply it consistently" still holds — just be
> aware that "reverse nothing, natural order" is the simpler of the two
> consistent choices, and the gogost reversals in D3/D4 cancel out rather than
> being load-bearing.

### D4. Inner-cipher word feeding inside `step`

The four ECB encryptions in `step` consume `H` in **8-byte words taken in
reverse byte order** (gogost hash.go:147-149 etc.: it feeds
`hin[31..24]`, then `hin[23..16]`, `hin[15..8]`, `hin[7..0]`, and writes the
8-byte cipher output back reversed into `out`). The engine equivalently calls
`gost_enc_with_key(c, Key, H + 0/8/16/24, S + ...)` (`gosthash.c:97,105,129,
137`). Same operation, different surface byte order — consistent with D3.

### D5. Key-schedule constants applied as `A(U) ⊕ Ci` — engine uses bit-NOT

RFC 5831 §5.1 builds `K3` via `U := A(U) ⊕ C3`. gogost XORs the literal C3
table (hash.go:177). The engine reaches the same value by **bit-flipping
specific `U` bytes** (`gosthash.c:108-123`, the run of `U[i] = ~U[i]`) — this
is exactly XOR with the C3 pattern expressed as `~` on the bytes where C3 has
`0xff`. Both produce identical `K3`. Document the C3 table form (D-spec above);
do not hand-translate the engine's NOT list.

### D6. Length block is bit-length, not byte-length

`L` accumulates the message length in **bits** (RFC 5831 §6). The engine does
`fin_len <<= 3` before writing it little-endian into the length block
(`gosthash.c:260-264`). gogost tracks `h.size` already in bits
(`h.size += BlockSize*8`, hash.go:237; partial `+= len*8`, hash.go:252) and
writes it big-endian into `block[24:]` (hash.go:259) — but because the whole
block is reversed convention-wise (D3), the on-wire word is little-endian.
A reimplementer must encode bit-length, not byte-length.

### D7. `Σ` is mod-2^256 addition, not XOR

The checksum uses carry-propagating addition (`add_blocks`, `gosthash.c:64-74`;
gogost `chkAdd` via `big.Int`, hash.go:225-232, subtracting `2^256` on overflow
hash.go:228). RFC 5831 §6 `(+)'` operator. Using XOR here is a common bug; it
matches for short inputs by luck and diverges once a carry occurs.

### D8. Destructive-`Sum` is avoided here, but watch your own port

gogost's `Sum` snapshots `size`, `chk`, `hsh` into locals (hash.go:247-249) so
the receiver is not mutated — `Sum` is non-destructive and may be followed by
more `Write`. The engine's `finish_hash` copies `H`/`S` into local buffers
first (`gosthash.c:247-248`) for the same reason. Replicate this snapshot or
you will corrupt a hash you intend to keep writing to. (Compare the gost28147
IMIT `Sum` gotcha in `CLAUDE.md` "gogost/v7 library gotchas".)

## Re-implementation checklist

Ordered, each step independently testable.

1. **GOST 28147-89 ECB encrypt** with the CryptoPro hash S-box
   (`sbox.go:93-102`). 32-round `gostcrypt` (`tmp/engine/gost89.c:278-...`).
   Test: encrypt a known block; or defer and validate transitively at step 7.
2. **Transform A** `A(X) = (x1⊕x2)||x4||x3||x2` over 64-bit words
   (RFC 5831 §5.1; `fA` hash.go:93-105 / `circle_xor8` `gosthash.c:42-50`).
3. **Transform P** byte permutation `φ` (§5.1; `fP` hash.go:107-115 /
   `swap_bytes` `gosthash.c:32-39`).
4. **Transform ψ** (`fChi` hash.go:117-123 / `transform_3` `gosthash.c:52-61`).
   Unit-test ψ against a hand-computed 32-byte vector.
5. **Compression χ** = key-gen (K1..K4 with C2/C3/C4) + 4 ECB encryptions +
   `ψ^61(H ⊕ ψ(M ⊕ ψ^12(S)))` (§5; `step` hash.go:135-223 /
   `hash_step` `gosthash.c:89-147`). Pin down your endianness convention here
   (deltas D3/D4) — get ONE block right.
6. **Block absorption**: per full block update `H = χ(M,H)`, `L += 256` bits,
   `Σ += M` mod 2^256 (carry, delta D7) (§6; hash.go:234-244 /
   `hash_block` `gosthash.c:199-233`).
7. **Finalization (non-empty)**: pad last fragment left with zeros, fold it in,
   then `H = χ(L,H)`, `H = χ(Σ,H)` (§6; `Sum` hash.go:246-267 /
   `finish_hash` `gosthash.c:240-269`). Snapshot state first (delta D8).
   Validate against KAT 1 ("a") and KAT 2 ("abc").
8. **Empty-input special case (delta D1)**: when total length is 0, insert the
   extra `χ(0^256, H)` BEFORE the length and checksum blocks to match
   gost-engine / Tarantool (`gosthash.c:257-258`). Validate against KAT 3
   engine value `3f25bc1f...`.
9. **Multi-block + Σ-carry**: validate against the 2100-byte engine vector
   `42e462ce...` (step 6/7 carry path) — `tmp/engine/tcl_tests/dgst.try:81`.
10. **Wire into HMAC**: confirm `HMAC-GOSTR341194` reproduces the TLS PRF used
    by suite `0x0081` (`tls/internal/suites/gost_suites_test.go:207`,
    `TestGOST_PRF_R341194`).

## Conformance & fuzz testing

The clean-room implementation is already complete. Conformance is verified by:

1. **In-package KATs** (`gostr341194/gostr341194_test.go`): table-driven
   `TestConformance` covers `"a"`, `"abc"`, `""` (engine value), `"message
   digest"`, `128*U`, `132*U`, lazy-dog/cog, and two standard-length messages.
   `TestStreamingMatchesOneShot` pins the `Write` buffering path (chunk sizes
   1/3/7/31/32/33/64/199). `TestSumNonDestructive` pins the delta-D8 snapshot
   contract.

2. **Differential parity** (`../gostcrypto-compat/parity/gostr341194/`): the
   compat module (GPL-quarantined) runs the clean-room `Sum` against the gogost
   oracle over 200+ random lengths and streaming split-points. The empty-input
   case is deliberately excluded from that diff (documented D1 divergence —
   clean-room matches engine, not gogost). Run with:
   ```sh
   ( cd ../gostcrypto-compat && go test ./parity/gostr341194/ -v )
   ```

**Note on the empty-input divergence (delta D1):** gogost and gost-engine
*disagree* for length-0 input. The clean-room MUST match the **engine** value
`3f25bc1f…` for Tarantool interop. The in-package KAT pins this; the compat
differential intentionally skips length-0.
