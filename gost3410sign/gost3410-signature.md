# GOST R 34.10-2001 / 34.10-2012 signature (sign + verify, key encoding)

GOST R 34.10 is the Russian-standard elliptic-curve digital signature
algorithm. It is structurally an EC-Schnorr/ElGamal variant (NOT ECDSA — the
`s` equation and the verification point combination differ), defined over a
prime-field short-Weierstrass curve with GOST-specific parameter sets.

*Intended implementer: a Sonnet-class coding agent — every constant, S-box, parameter table, and edge case is inlined so this primitive can be built without consulting gogost or external specs.*

Two generations exist, sharing the *same* point arithmetic and the *same*
sign/verify equations; they differ only in the companion hash and the
supported key sizes:

| Generation | Standard | RFC | Hash | Sizes |
|---|---|---|---|---|
| GOST R 34.10-2001 | GOST R 34.10-2001 | RFC 5832 | GOST R 34.11-94 | 256-bit only |
| GOST R 34.10-2012 | GOST R 34.10-2012 | RFC 7091 | Streebog-256 / Streebog-512 | 256-bit and 512-bit |

Because the sign/verify math is identical across generations, ONE
implementation covers both — the only generation-dependent choice is which
digest feeds in and how wide the curve is (32-byte vs 64-byte coordinates).

### Where the repo uses it

- **x509gost certificate-chain verification** — `x509gost/verify.go:204` calls
  `gost.VerifyDigestOnCurve(curve, parent.PubKeyRaw, digest, sig)`. The parent
  cert's curve is resolved from its parameter OID at `x509gost/verify.go:196`,
  and the digest is chosen by the child's signature algorithm
  (`verify.go:180-188`): GOST R 34.11-94 for `AlgoR341001`, Streebog-256 for
  `AlgoR341012_256`, Streebog-512 for `AlgoR341012_512`. Every GOST TLS
  certificate flows through this path.
- **TLS key-exchange** — `tls/internal/ke/vkogost.go` calls
  `PublicKeyRawFromPrivate` to derive the ephemeral client public key for the
  VKO key agreement (the signature primitive's `d·P` point multiplication is
  shared with VKO).

This primitive is NOT directly in the record/MAC hot path — it fires once per
certificate during the handshake. TLS suite IDs that ultimately depend on it:
`0x0081` (GOST2001-GOST89-GOST89, RFC 4357/legacy), `0xFF85`
(GOST2012-GOST8912-GOST8912, RFC 9189), and the RFC 9367 suites
`0xC100`/`0xC101` — all carry GOST-signed server certificates verified here.

## Specification

All numbered references below are to RFC 7091 (2012); the 2001 step numbering
in RFC 5832 is identical. Source-of-truth implementations: gogost
`third_party/gogost/gost3410/{private,public,curve}.go` and gost-engine
`tmp/engine/gost_ec_sign.c`.

### Domain parameters (the curve)

A GOST R 34.10 curve in canonical short-Weierstrass form is
`y² = x³ + a·x + b (mod p)` with:

- `p` — characteristic of the prime field. 256-bit curves: `p.BitLen() <= 256`,
  coordinate size 32 bytes. 512-bit curves: `p.BitLen() > 256`, coordinate
  size 64 bytes. (`third_party/gogost/gost3410/utils.go:36` `pointSize`.)
- `a`, `b` — curve coefficients.
- `q` — order of the cyclic subgroup generated by the base point. Both `r`
  and `s` live mod `q`. (gogost field `Curve.Q`.)
- `(x, y)` — base point `P` (gogost fields `Curve.X`, `Curve.Y`).
- `co` — cofactor (1 for the 256-bit/512-bit prime-order subgroups in use; the
  test sweep at `internal/gost/curve_sign_sweep_test.go:130` asserts 1 or 4).

The curves the repo resolves by OID (`internal/gost/primitives_gost.go:53`):

### Point arithmetic

Affine short-Weierstrass over `GF(p)`. gogost implements:

- **Point addition / doubling** — `Curve.add`,
  `third_party/gogost/gost3410/curve.go:108`. Standard slope formula:
  doubling slope `t = (3·x² + a) / (2·y) mod p`; addition slope
  `t = (y₂−y₁) / (x₂−x₁) mod p`. Result
  `x₃ = t² − x₁ − x₂`, `y₃ = t·(x₁ − x₃) − y₁`, all mod p, normalized
  non-negative via `Curve.pos`.
- **Scalar multiplication `k·P`** — `Curve.Exp`,
  `third_party/gogost/gost3410/curve.go:144`. Left-to-right double-and-add
  over the bits of `k−1` (it seeds the accumulator with `P` and iterates from
  `k−1`, so a zero scalar is rejected: `curve.go:145`). NOTE: this is a
  non-constant-time, branch-on-bit ladder. A reimplementation MAY use a
  constant-time ladder; the *output* must match. There is no projective-coord
  subtlety to match — only the final affine `(x, y)`.

gost-engine routes the multiply through OpenSSL's `EC_POINT_mul`
(`tmp/engine/gost_ec_sign.c:531`); same affine result.

### Signature generation (RFC 7091 §6.1)

Input: message hash `H = h(M)` (32 or 64 bytes), private key `d` (integer
`0 < d < q`).

1. **§6.1 step 2** — interpret `H` as integer `alpha` and reduce:
   `e = alpha mod q`. *"If e = 0, then assign e = 1."*
2. **§6.1 step 3** — generate random `k`, `0 < k < q`.
3. **§6.1 step 4** — `C = k·P`; `r = x_C mod q`. If `r = 0`, go to step 2.
4. **§6.1 step 5** — `s = (r·d + k·e) mod q`. If `s = 0`, go to step 2.
5. **§6.1 step 6** — output `zeta = (R || S)`, binary vectors of `r` and `s`.

gogost: `PrivateKey.SignDigest`,
`third_party/gogost/gost3410/private.go:99-140`. gost-engine:
`gost_ec_sign`, `tmp/engine/gost_ec_sign.c:182-319` (the
`s = r·priv + k·e` step is at lines 290-292).

### Signature verification (RFC 7091 §6.2)

Input: `H`, public key point `Q`, signature `(r, s)`.

1. **§6.2 step 1** — reject unless `0 < r < q` and `0 < s < q`.
2. **§6.2 step 3** — `e = alpha mod q`; if `e = 0`, `e = 1`.
3. **§6.2 step 4** — `v = e⁻¹ mod q`.
4. **§6.2 step 5** — `z1 = s·v mod q`; `z2 = −r·v mod q` (i.e. `(q − r·v) mod q`).
5. **§6.2 step 6** — `C = z1·P + z2·Q`; `R = x_C mod q`.
6. **§6.2 step 7** — valid iff `R == r`.

gogost: `PublicKey.VerifyDigest`,
`third_party/gogost/gost3410/public.go:89-143`. gost-engine:
`gost_ec_verify`, `tmp/engine/gost_ec_sign.c:325-...` (`z2` computed as
`tmp = order − sig_r; z2 = tmp·v`, lines 390-393).

### Key encoding — Raw() byte layout (the part most likely to bite you)

GOST encodes integers **little-endian** on the wire, the opposite of nearly
every other EC stack. The "Raw" forms used throughout this repo are LE.

**Private key** (`prv.Raw()` = `RawLE`,
`third_party/gogost/gost3410/private.go:75`): `PointSize` bytes,
little-endian. 32 bytes for 256-bit, 64 bytes for 512-bit. Parsing
(`NewPrivateKeyLE`, `private.go:32`) byte-reverses the input, then reduces
mod `q`. A zero key is rejected.

**Public key** (`pub.Raw()` = `RawLE`,
`third_party/gogost/gost3410/public.go:65`): `2 · PointSize` bytes — the
concatenation `LE(X) || LE(Y)`:

```
256-bit pubkey (64 bytes total):  [ LE(X) : 32 ] [ LE(Y) : 32 ]
512-bit pubkey (128 bytes total): [ LE(X) : 64 ] [ LE(Y) : 64 ]
```

WATCH THE CONSTRUCTION ORDER. `RawLE` builds `pad(Y) || pad(X)` in
BIG-endian and then reverses the WHOLE buffer (`public.go:67-71`). Reversing
`BE(Y)||BE(X)` yields `LE(X)||LE(Y)` — so the final layout is X-first,
Y-second, each little-endian. Do not copy the intermediate Y-first ordering;
only the post-reverse layout is observable. `NewPublicKeyLE` (`public.go:30`)
inverts exactly this: reverse the whole 2·PointSize buffer, then the first
half is `Y` and the second half is `X` (because the reverse swaps halves
back).

**Signature** (`r||s` packed, `third_party/gogost/gost3410/private.go:135-139`):
`2 · PointSize` bytes laid out **`pad(s) || pad(r)`**, i.e. `s` FIRST, `r`
SECOND, each big-endian-padded inside its half. Verify reads it the same way
(`public.go:94-95`: `s = sig[:pointSize]`, `r = sig[pointSize:]`).

```
256-bit signature (64 bytes):  [ s : 32 BE ] [ r : 32 BE ]
512-bit signature (128 bytes): [ s : 64 BE ] [ r : 64 BE ]
```

Subtle trap: the signature halves are stored **big-endian** within each
half (plain `pad(s.Bytes())` / `pad(r.Bytes())`, no reverse — `private.go:137`),
whereas the public/private *keys* are little-endian. Do not assume "GOST = LE
everywhere." The TLS/X.509 transport wraps `s||r` differently again
(`PrivateKeyReverseDigestAndSignature`, `private.go:176`, reverses the whole
signature) — but the `internal/gost` wrappers used by this repo
(`VerifyDigestOnCurve`/`SignDigestOnCurve`) use the plain non-reversed
`s||r` form. Match that.

## RFC ↔ implementation deltas

Every point where gogost / gost-engine deviate from, reinterpret, or
under-specify the RFC. Cite both the RFC and the source line.

1. **Digest-as-integer: read the bytes you are handed big-endian; do NOT
   byte-reverse.** gogost reads the digest with `e = bytes2big(digest)`, where
   `bytes2big` is `big.Int.SetBytes` (BIG-endian) — `private.go:100` (sign),
   `public.go:102` (verify), `utils.go:22`. The `internal/gost` wrappers used
   by this repo therefore consume the digest big-endian, and the repo feeds it
   to them un-reversed: `x509gost/verify.go:201-204` ("Digest is verified
   as-is … without byte-order massaging"), confirmed against Tarantool-EE
   3.5.0 certs by `TestTarantoolEE_Ping_GOST_Pure`. The §A.1 KAT digest
   `2dfbc1b3…2b043ee5`, read big-endian and reduced mod q, equals RFC 7091
   §7.2 `e = 0x2DFBC1B3…2B043EE5` exactly (reading it little-endian gives a
   different, wrong `e` and the §7 signature does NOT verify).

   Note the gost-engine internals differ at the BN-conversion boundary:
   `md = BN_lebin2bn(dgst, dlen, ...)` (`tmp/engine/gost_ec_sign.c:206` sign,
   `:375` verify) — a *little*-endian load. This is NOT a contradiction:
   gost-engine and gogost simply disagree on which side of the conversion the
   byte-reversal lives, and both ultimately operate on the same integer for the
   same on-the-wire digest. A clean-room implementation behind the
   `internal/gost` surface must match gogost (big-endian `SetBytes` of the
   bytes passed in) so the §A.1 vector and the repo's un-reversed call sites
   keep verifying. RFC 7091 §5.3 specifies `H` with the high-order bits on the
   left, i.e. a big-endian read — consistent with the gogost/internal-gost
   convention pinned here.

2. **`e = 0 → e = 1` substitution.** RFC 7091 §6.1/§6.2 mandate it. Easy to
   forget; gogost `private.go:102-104`, `public.go:104-106`; gost-engine
   `gost_ec_sign.c:239-241`, `:386`. Without it, a digest that reduces to 0
   mod q makes `v = e⁻¹` undefined.

3. **`z2` sign handling.** RFC says `z2 = −r·v mod q`. gost-engine computes
   `q − r` first then multiplies (`gost_ec_sign.c:392-393`). gogost computes
   `z2 = r·v mod q` then `z2 = q − z2` (`public.go:113-115`). Same result;
   both avoid a negative intermediate. A naive `(-r*v) mod q` in `math/big`
   yields a negative — normalize.

4. **Signature half-ordering is `s||r`, NOT `r||s`.** Despite the RFC writing
   the signature as `R||S` (r first) in §6.1 step 6, the packed byte form in
   this repo is `s` then `r` (`private.go:136-139`, `public.go:94-95`). This
   is a real interop landmine: get it backwards and verification silently
   fails. (The X.509 ASN.1 envelope and the
   `PrivateKeyReverseDigestAndSignature` transform reorder/reverse again —
   but the `internal/gost` raw form is `s||r`.)

5. **Public-key Raw is `LE(X)||LE(Y)` via a whole-buffer reverse.** See key
   encoding above. The intermediate big-endian `Y||X` at `public.go:67-70`
   is NOT the observable layout — only after the reverse. Reimplementers who
   reverse each coordinate independently (instead of the whole buffer) get
   the same answer; reimplementers who forget that the *parse* path also
   swaps halves (`public.go:36-43`) will transpose X and Y.

6. **Scalar `k`/`d` reduced mod q on load, not validated as `< q`.**
   `NewPrivateKeyLE` does `k.Mod(k, q)` (`private.go:45`); `SignDigest`
   reduces `k` likewise (`private.go:116`) and retries on `k ≡ 0`. RFC §6.1
   step 3 says "generate `k` with `0 < k < q`"; gogost achieves this by
   reduce-then-reject-zero rather than rejection sampling. gost-engine uses
   `BN_rand_range(k, order)` (`gost_ec_sign.c:251`) — true rejection
   sampling. Both are spec-compliant; the *distribution* differs only
   negligibly. For deterministic test signing the repo feeds a fixed reader
   (`curve_sign_sweep_test.go:60`).

7. **`r==0` and `s==0` retry loops.** RFC §6.1 steps 4/5 require regenerating
   `k`. gogost `goto Retry` (`private.go:125,132`); gost-engine nested
   `do…while` (`gost_ec_sign.c:249-297`). Don't skip these — omitting them
   leaks a tiny bias and can emit an invalid signature.

8. **Verify range check is strict on BOTH ends.** `0 < r,s < q`.
   gogost rejects `r ≤ 0 || r ≥ q || s ≤ 0 || s ≥ q` (`public.go:96-100`);
   gost-engine the equivalent (`gost_ec_sign.c:367-369`). A value equal to
   `q` must be rejected, not reduced.

9. **No S-box / key-meshing / counter quirks here.** Of the three known
   gogost↔gost-engine divergences logged in `TODO.md` (S-box row order, R
   34.11-94 empty-input finalization, CryptoPro key meshing), NONE touch this
   primitive — they live in the 28147/34.11-94 layers. The R 34.11-94
   empty-input divergence (`TODO.md` line 7) only matters for the *companion
   hash* of 2001 certs, and only for an empty message, which never occurs for
   a real TBSCertificate. So a signature reimplementation can ignore all
   three. (Documented here so a reimplementer doesn't go hunting.)

10. **512-bit sign is unexposed in the Phase-1 wrappers but fully supported in
    the curve layer.** `R341012Sign`/`Verify` hardcode the 256-bit
    TestParamSet (`primitives_gost.go:190,202`), but `SignDigestOnCurve` /
    `VerifyDigestOnCurve` are curve-generic and the sweep test
    (`curve_sign_sweep_test.go:42-44`) exercises all three 512-bit curves
    with 64-byte digests. A reimplementation must size buffers off
    `PointSize`, never a 32 constant.

## Re-implementation checklist

Each step is independently testable against a vector above.

1. **Bignum + field ops.** Use `math/big`. Implement `mod` that normalizes to
   `[0, p)` (Go's `big.Int.Mod` already returns non-negative for positive
   modulus). Test: trivial.
2. **Curve type + parameter table.** Encode `p, a, b, q, (x,y), co` for all 10
   OIDs (RFC 7836 §3 / `tmp/engine/gost_params.c`). Test: `Contains(P)` true;
   the `(q+1)·P == P` invariant (`curve_sign_sweep_test.go:135`).
3. **Point add/double** (`y² = x³ + ax + b`). Test: `2·P` then verify
   `Contains`; `3·P` two ways (`P+P+P` vs double-then-add).
4. **Scalar multiply `k·P`** (any correct ladder; constant-time encouraged).
   Test: `(q+3)·P == 3·P` (`curve_sign_sweep_test.go:144-153`).
5. **Private-key parse/marshal (LE).** `NewPrivateKeyLE`: reverse bytes,
   `SetBytes`, `mod q`, reject zero. Test: round-trip `d_LE` from §A.1.
6. **Public-key derive + Raw (LE X||Y).** `Q = d·P`; `Raw = LE(X)||LE(Y)`.
   Test: assert against the §A.1 `X_LE`/`Y_LE` (matches
   `primitives_pkey_vectors_test.go:127-134`).
7. **Digest→integer.** Read the digest BIG-endian — `big.Int.SetBytes` of the
   bytes as-handed, matching gogost's `bytes2big` (delta #1); do NOT
   byte-reverse. Then `e = alpha mod q`, `e==0 ⇒ e=1`. Test: against §A.1
   where the digest `2dfbc1b3…2b043ee5` read big-endian ⇒ `e = 0x2DFBC1B3…`.
8. **Verify** (no RNG, fully deterministic — do this BEFORE sign). Range-check
   `r,s ∈ (0,q)`; `v=e⁻¹`; `z1=s·v`, `z2=(q−r·v) mod q`; `C=z1·P+z2·Q`;
   `R=x_C mod q`; accept iff `R==r`. Parse signature as `s||r`. Test: the
   §A.1 `s||r` verifies `true`; tampered digest `false`.
9. **Sign.** `e` as in step 7; loop: random `k∈(0,q)`, `C=k·P`, `r=x_C mod q`
   (retry if 0), `s=(r·d+k·e) mod q` (retry if 0); emit `s||r`. Test:
   sign-then-verify round-trip on every curve
   (`curve_sign_sweep_test.go`); with a fixed RNG feeding §A.1's `k`,
   reproduce the exact `r`,`s` above.
10. **512-bit sizing.** Drive everything off `PointSize` (32 vs 64); never
    hardcode 32. Test: 512-bit rows of `TestCurveSignVerify_AllCurves`.
11. **Wire the wrappers.** Implement `VerifyDigestOnCurve`,
    `SignDigestOnCurve`, `PublicKeyRawFromPrivate`, `CurveByOID`,
    `R342001Verify`, `R341012Sign/Verify` with identical signatures, then run
    the existing `internal/gost` test suite under `-tags gost` unchanged.

## Conformance & fuzz testing

Once the clean-room sign/verify is wired behind the same `internal/gost`
surface, prove parity by differential testing against the **in-repo gogost
reference** (`SignDigestOnCurve`/`VerifyDigestOnCurve`,
`exports_gost.go:115,148`; and `R341012Sign`/`R341012Verify`,
`primitives_gost.go:189,201`). Two properties matter because signing is
**randomized** (the nonce `k` makes `Sign` non-deterministic — you cannot
KAT the signature bytes for a random `k`): (1) the round-trip
`Verify(pub, digest, Sign(d, digest)) == true` for every fresh `(d, digest)`,
and (2) **cross-verification** — a signature produced by gogost must verify
under the clean-room verifier and vice versa, since verify *is*
deterministic. Fuzz a random 32-byte private key plus a random 32-byte digest
on a fixed 256-bit curve (the RFC 7091 TestParamSet, `GOST2001TestParamSetCurve()`,
`exports_gost.go:161`); the only KAT-able pins are the §7 vector below
(deterministic verify) and the public-key derivation.

### Table-driven KAT (deterministic surface)

Seeded with the exact pinned hex from the Test vectors section above (the
RFC 7091 §7 256-bit TestParamSet `s||r` signature and `X_LE||Y_LE` public
key). Verification and pubkey-derivation are deterministic, so they pin
directly; the signature `s||r` is only asserted to *verify*, never to be
re-produced (random `k`). `mynew` is the clean-room package; `gost` is the
in-repo gogost-backed reference.

> **Skeleton caveat — `mynew.GOST2001TestParamSetCurve()` is illustrative,
> not a real clean-room symbol.** `GOST2001TestParamSetCurve()` exists only on
> the gogost-backed oracle (`internal/gost/exports_gost.go:161`). The actual
> clean-room curve layer (`gost3410curves`) exposes only
> `CurveByOID`, and the RFC 7091 §7.1 TestParamSet is **OID-less** (a test-only
> curve), so it is *not* resolvable that way. A reimplementer must build the
> TestParamSet `Curve` literally from the §7.1 hex above (`p, a, b, q, X, Y`),
> as the `mynew.…` placeholder in the skeleton below stands in for. Likewise
> the skeleton's `mynew.{PublicKeyRawFromPrivate,SignDigestOnCurve,
> VerifyDigestOnCurve}` names mirror the oracle wrapper surface; the
> task-mandated clean-room entrypoints are `SignDigest(c, prv, digest, k)` /
> `VerifyDigest(c, pubRaw, digest, sig)` / `PublicKeyRaw(c, prv)` (different
> signatures — a `k []byte` nonce read big-endian rather than an `io.Reader`,
> no `error` return). Adapt the calls accordingly.

```go
//go:build gost

package conformance

import (
	"bytes"
	"encoding/hex"
	"testing"

	gost "go.bigb.es/tlsdialer/internal/gost" // in-repo gogost-backed reference
	mynew "yourpkg/gost"            // clean-room re-implementation under test
)

func mustHex(t *testing.T, s string) []byte {
	t.Helper()
	b, err := hex.DecodeString(s)
	if err != nil {
		t.Fatalf("bad hex %q: %v", s, err)
	}
	return b
}

func TestGost3410Conformance(t *testing.T) {
	// RFC 7091 §7 / TestParamSet — see "Test vectors" above.
	const (
		prvLE  = "283bec9198ce191dee7e39491f96601bc1729ad39d35ed10beb99b78de9a927a"
		digBE  = "2dfbc1b372d89a1188c09c52e0eec61fce52032ab1022e8e67ece6672b043ee5" // read big-endian; fed as-is
		pubXLE = "0bd86fe5d8db89668f789b4e1dba8585c5508b45ec5b59d8906ddb70e2492b7f"
		pubYLE = "da77ff871a10fbdf2766d293c5d164afbb3c7b973a41c885d11d70d689b4f126"
		// signature s||r, big-endian within each 32-byte half.
		sigSR = "01456c64ba4642a1653c235a98a60249bcd6d3f746b631df928014f6c5bf9c40" +
			"41aa28d2f1ab148280cd9ed56feda41974053554a42767b83ad043fd39dc0493"
	)
	prv := mustHex(t, prvLE)
	dig := mustHex(t, digBE)
	wantPub := append(mustHex(t, pubXLE), mustHex(t, pubYLE)...)
	sig := mustHex(t, sigSR)

	refCurve := gost.GOST2001TestParamSetCurve()
	newCurve := mynew.GOST2001TestParamSetCurve()

	// 1. public-key derivation (deterministic) — both must equal the pin.
	refPub, err := gost.PublicKeyRawFromPrivate(refCurve, prv)
	if err != nil {
		t.Fatalf("ref PublicKeyRawFromPrivate: %v", err)
	}
	newPub, err := mynew.PublicKeyRawFromPrivate(newCurve, prv)
	if err != nil {
		t.Fatalf("new PublicKeyRawFromPrivate: %v", err)
	}
	if !bytes.Equal(refPub, wantPub) {
		t.Fatalf("ref pub %x != pin %x", refPub, wantPub)
	}
	if !bytes.Equal(newPub, wantPub) {
		t.Fatalf("new pub %x != pin %x", newPub, wantPub)
	}

	// 2. verify the pinned s||r signature (deterministic) under both impls.
	okRef, err := gost.VerifyDigestOnCurve(refCurve, wantPub, dig, sig)
	if err != nil || !okRef {
		t.Fatalf("ref verify pinned sig: ok=%v err=%v", okRef, err)
	}
	okNew, err := mynew.VerifyDigestOnCurve(newCurve, wantPub, dig, sig)
	if err != nil || !okNew {
		t.Fatalf("new verify pinned sig: ok=%v err=%v", okNew, err)
	}

	// 3. tamper: flip one digest byte → both must reject.
	bad := append([]byte(nil), dig...)
	bad[0] ^= 0x01
	if ok, _ := mynew.VerifyDigestOnCurve(newCurve, wantPub, bad, sig); ok {
		t.Fatalf("new verify accepted tampered digest")
	}
}
```

### Fuzz harness (round-trip + cross-verify)

`SignDigestOnCurve` takes an `io.Reader` nonce source
(`exports_gost.go:148`), so feed a deterministic reader derived from the
fuzz input to keep replays reproducible. The harness normalizes the random
`[]byte` into the fixed 32-byte private key and 32-byte digest this curve
needs, then signs with each impl and cross-verifies under the other.

```go
//go:build gost

package conformance

import (
	"bytes"
	"testing"

	gost "go.bigb.es/tlsdialer/internal/gost"
	mynew "yourpkg/gost"
)

func FuzzGost3410Conformance(f *testing.F) {
	// Seed from the KAT inputs (prvLE, digBE concatenated = 64 bytes).
	f.Add(mustHexF("283bec9198ce191dee7e39491f96601bc1729ad39d35ed10beb99b78de9a927a" +
		"2dfbc1b372d89a1188c09c52e0eec61fce52032ab1022e8e67ece6672b043ee5"))

	refCurve := gost.GOST2001TestParamSetCurve()
	newCurve := mynew.GOST2001TestParamSetCurve()

	f.Fuzz(func(t *testing.T, raw []byte) {
		// Normalize into fixed-size 32B private key + 32B digest.
		buf := make([]byte, 64)
		copy(buf, raw)
		prv := buf[:32]
		dig := buf[32:]
		// A LE private key that reduces to 0 mod q is rejected on load; skip.
		if isZero(prv) {
			t.Skip()
		}

		// Deterministic nonce reader so a failing input replays identically.
		newRnd := func() *bytes.Reader { return bytes.NewReader(append(buf, buf...)) }
		refRnd := func() *bytes.Reader { return bytes.NewReader(append(buf, buf...)) }

		refPub, err := gost.PublicKeyRawFromPrivate(refCurve, prv)
		if err != nil {
			t.Skip() // invalid key for this curve; not a parity bug.
		}
		newPub, err := mynew.PublicKeyRawFromPrivate(newCurve, prv)
		if err != nil {
			t.Fatalf("new derive failed where ref succeeded: %v", err)
		}
		if !bytes.Equal(refPub, newPub) {
			t.Fatalf("pub mismatch: ref=%x new=%x", refPub, newPub)
		}

		// Sign with each impl (randomized k); verify each sig under BOTH.
		refSig, err := gost.SignDigestOnCurve(refCurve, prv, dig, refRnd())
		if err != nil {
			t.Fatalf("ref sign: %v", err)
		}
		newSig, err := mynew.SignDigestOnCurve(newCurve, prv, dig, newRnd())
		if err != nil {
			t.Fatalf("new sign: %v", err)
		}

		// Cross-verify: ref's sig under new, new's sig under ref, plus self.
		check := func(name string, ok bool, err error) {
			if err != nil || !ok {
				t.Fatalf("%s: ok=%v err=%v (prv=%x dig=%x)", name, ok, err, prv, dig)
			}
		}
		ok, err := mynew.VerifyDigestOnCurve(newCurve, refPub, dig, refSig)
		check("new verifies ref sig", ok, err)
		ok, err = gost.VerifyDigestOnCurve(refCurve, refPub, dig, newSig)
		check("ref verifies new sig", ok, err)
		ok, err = mynew.VerifyDigestOnCurve(newCurve, refPub, dig, newSig)
		check("new verifies new sig", ok, err)
	})
}
```

Helpers (`mustHexF`, `isZero`) are trivial: `mustHexF` is `hex.DecodeString`
with a panic on error (fuzz seeds are compile-time constants), and `isZero`
returns true iff every byte is `0x00`. Both 256-bit and 512-bit curves can be
swept by replacing the single fixed curve with a table over `CurveByOID`
(`primitives_gost.go:53`) and sizing `prv`/`dig` off `curve.PointSize()`
(`exports_gost.go:129`) — 32 vs 64 bytes — exactly as
`curve_sign_sweep_test.go:42` does.

> No gost-engine CLI oracle is needed here: the signature primitive has a
> first-class gogost API (`gost3410`), so the in-repo wrappers are the
> reference. The CLI-oracle pattern from CLAUDE.md (`openssl dgst -engine
> gost …`) is only required for primitives with no gogost surface (OMAC,
> CTR-ACPKM, KEG, KExp15, KeyWrap) — not for GOST R 34.10.
